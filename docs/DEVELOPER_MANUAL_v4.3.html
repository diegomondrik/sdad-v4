1 · What SDAD is and why it exists

SDAD (Spec-Driven AI Development) is a methodology for building software with Claude Code as the primary tool. Its founding rule fits in one line:

🔒 Golden rule Production code is never written before the developer approves a Spec. The only exception is $docfinal, which documents already-built projects retroactively.

Why? Because the main risk of AI-assisted development is not that the model writes code badly — it is that it writes the wrong code fast. Without an agreed specification, every session improvises decisions, scope drifts, and by day three nobody knows why the system does what it does. SDAD inverts the order: first we agree on what and why (Spec), then the model executes how (Build), and every increment closes with quality assurance (QA) and a decision record.

Everything lives in versioned files in the repo — not in chat memory:

FileWhat it holds
CLAUDE.mdThe methodology rules (the "brain" Claude Code reads every session)
SPEC.mdThe approved specification: the project contract
DECISIONS.mdEvery architectural decision, with date, model and rationale. Entries marked [LOCK] are never reopened
LESSON_LIBRARY.mdTransferable lessons across projects (L-XX)
.claude/Specialist skills, isolated agents and session hooks

2 · The phases: the full cycle

0 · Context→ 1 · Requirements ($spec)→ 2 · Spec ($specout)→ 3 · Build ($build)→ 4 · QA ($qa)→ 5 · Docs ($doc)

Phase 0 — Context analysis

Automatic at session start. Claude reads the actual repo (it never asks what it can infer), detects UI presence, recommends a compliance tier, and surfaces 2–3 relevant lessons from the Lesson Library. If the project went more than 30 days without a $build, it triggers $verify audit to check for outdated dependencies.

Phase 1 — Guided requirements ($spec)

Claude asks one question at a time, always with a proposed default — never a questionnaire. It covers: scope, user flows, data model, integrations, business rules, performance, security, compliance tier and testing. The tier question is mandatory and never skipped. The project language question (English/Spanish) comes first — see section 6.

💡 Document ingestion (MarkItDown) When the requirements arrive as binary documents — PDF, Word, Excel, PowerPoint, Outlook messages — Claude converts them to Markdown first with Microsoft MarkItDown (pip install 'markitdown[all]' → markitdown file.pdf -o file.md) and reads the Markdown: it preserves headings, tables and lists, and consumes far fewer tokens. Applies to $spec, §A client diagnosis and $docfinal. Converted copies live in .sdad/ingest/. Local trusted files only — MarkItDown performs I/O with the privileges of the process.

Phase 2 — Spec generation ($specout)

Generates the full document (§1 Vision … §13 AI Authorship Log) and writes it directly to SPEC.md, then asks for your approval. No approval, no $build.

Phase 3 — Guided development ($build)

Work happens in vertical increments: complete slices of functionality with their tests. Before each increment, Claude announces which files it touches, which tests it will run, which docs it updates, and which model/effort it recommends (see routing). After writing the code: it runs the real tests, runs $qa, records the decision in DECISIONS.md and updates §13 of the Spec.

Phase 4 — QA ($qa)

Five layers, in priority order: 🔐 Security → 🏗️ Structure → ⚡ Efficiency → ✅ Best Practices → 🟠 Platform (Pyplan only). Findings are numbered H-XX with severity P0/P1/P2.

🔒 Security gate Automatic $qa never touches security, compliance or Spec deviations without explicit human approval. P0 findings are flagged 🚨 in any phase.

Phase 5 — Documentation ($doc)

Automatically delegated to a sub-agent (it does not consume your context) and written to /docs: README, API reference, architecture document, compliance summary.

5 · Model & effort routing new in v4.3

Not every phase needs the same model. Designing a Spec demands the best reasoning available; executing an already-specified increment does not. v4.3 formalizes this with a per-phase routing table using model-agnostic tiers — so the table survives every new model release:

TierMeansToday, for example
FRONTIERBest reasoning model availablefable, opus
STANDARDCost/capability balancesonnet
ECONOMYFastest and cheapesthaiku
PhaseModel · EffortWhy
$spec / $specoutFRONTIER · highOpen decisions, requirements design
$build (per increment)STANDARD · low — or FRONTIER · high when the increment carries risk or an open decisionExecuting what is specified is cheap; deciding is not
$qa incrementalSTANDARD · mediumReview bounded to one increment
$qa full / $QA / $docfinalFRONTIER · highWhole-codebase judgment
$verify / $docECONOMY–STANDARD · lowMechanical, delegable
$agent review/auditPinned in frontmatter: opus · highVía B — the sub-agent brings its own model
$agent testPinned in frontmatter: sonnet · mediumContract-driven generation
$pause / $lesson / $flowCurrent model · lowNever worth switching for these
⚠️ Important: the session never switches model by itself Claude Code does not auto-switch model or effort. The mechanism is: at the start of each phase, Claude emits the line 🧠 MODEL: [recommended] · effort [level]; if your active session differs, it flags it and you run /model and /effort. Only $build blocks code writing on a mismatch — all other phases flag once and continue.

6 · Project language new in v4.3

The methodology repo and its rules are in English, but each project chooses its working language. The very first $spec question asks: English or Spanish? The answer is locked in the project declaration (PROJECT_LANGUAGE: en | es) and from then on everything follows it: the questions Claude asks, SPEC.md, DECISIONS.md entries, lesson candidates, QA reports and generated documentation. Code identifiers and comments stay in English regardless (industry convention); the language setting governs documents and interaction.

7 · SDAD for Pyplan

When a project declares PROJECT_PLATFORM: pyplan in CLAUDE.md, SDAD extends with platform-specific Spec sections, skills and checklists. Nothing in the base cycle changes — layers are added on top.

Extra Spec sections (prepended before §1)

SectionWhat it documentsGate?
§0 Platform ContextPyplan version, workspace, permissions, data types, conventionsNo
§A Data ArchitectureClient diagnosis, architecture decision, data contract per sourceYes — blocks $build until approved
§B Discovery LogInitialized empty; records data deltas found during $buildNo
§D MCP Tools CatalogOnly if the project exposes @mcp_tool nodes: name, description, typed parameters, serializable returnYes — when present, blocks $build until approved
💡 The logic behind the gates In Pyplan projects the most expensive mistake is building the model on data that is not what was assumed. §A forces the data contract to be agreed before building — the same way §9 Security blocks the build on Tier 3 projects.

Data deltas: the discovery protocol

During $build, the client's real data usually deviates from §A. SDAD distinguishes two cases:

Delta typeExamplesWhat happens
SmallFormat error, nulls, field with another name, unexpected volumeResolved in the node, recorded in §B, $build continues
StructuralThe data does not exist, wrong granularity, the source is something else entirely🛑 $build pauses. A data gap report is generated for the consultant to take to the client. A workaround is never improvised

Pyplan increment checklist

Every Pyplan increment closes by verifying: all new nodes have result= assigned and calculate without error; no circular dependencies; read-node sources match the §A data contract; indexes are synchronized; inputs have validations; visualization components display calculated data (not empty); and any deltas found were recorded in §B. If the increment touches an HTML interface, the HTML interface surface block also runs (see below).

HTML interfaces

Pyplan supports two interface types. A component interface is the classic grid of draggable components, built manually. An HTML interface is a single, self-contained web page (HTML, CSS and JavaScript) that replaces the grid entirely and talks to the model through Python callbacks. HTML interfaces are designed to be created by AI agents — Pyplan's own agents or external AI clients via Pyplan MCP — and they are the default type whenever an agent builds a screen.

The anatomy is two parts stored together: the HTML page, and a Python class HTMLInterface(HTMLInterfaceBase) whose @callback methods are the only ones the page can call. The page reaches Python through the built-in window.pyplan bridge (callback, on, toast, import). Getters return JSON-friendly data; mutators change the model via set_input / set_form_values and return the stale-widget list from get_nodes_to_refresh(). Headings backed by nodes carry data-pyplan-nodes, which links the number on screen back to the influence diagram.

🔒 Sandbox — the rules that matter The page runs in a sandboxed frame with an isolated origin: no cookies, no localStorage, no <iframe> embeds, no authenticated requests outside window.pyplan.callback. Consequence: state must persist through the model via callbacks. External scripts, styles, fonts and images are allowed (Pyplan proxies them same-origin); dynamic JS modules load via window.pyplan.import(url).
💡 How SDAD treats them The methodology does not change. An AI-built HTML interface is an increment under the Build-via-AI guardrails: approved Spec, announcement, your approval, QA, DECISIONS.md entry. The increment closes with the HTML interface surface checklist (CLAUDE.md), and $qa Layer 5 re-verifies the same rules. $spec asks which screens need a fully custom layout so the interface type is recorded per screen in §3/§5.

MCP in Pyplan (@mcp_tool nodes)

Pyplan can expose nodes as MCP tools that other AI clients invoke. SDAD treats that surface with extra discipline because it is an outward-facing contract:

  • A docstring that explains the business action precisely — an external LLM must be able to invoke it correctly just by reading it.
  • All parameters typed with Annotated[type, 'description'] — nothing untyped.
  • Serializable return values: no raw xarray, no DataFrames without .to_dict().
  • result = _fn (the function is assigned, not called).
  • QA Layer 1 verifies: OAuth tokens never in logs or results (P0), validated parameters (P1), minimum scope (P2).
⚠️ Build-via-AI The Pyplan MCP lets an AI client modify the application directly. SDAD allows it with guardrails: every MCP-driven modification is an increment — it requires an approved Spec, a prior announcement, your approval, QA and a DECISIONS.md entry. The MCP does not bypass the Spec gate. Also, the Pyplan MCP server is v1: it is documented as an external dependency in §7 and pinning the Pyplan version in §5 is recommended.

8 · Day-to-day usage

Installation (once per machine + once per project)

# Windows (PowerShell) — from the project repo:
$install = (Invoke-WebRequest -Uri "https://raw.githubusercontent.com/diegomondrik/sdad-v4/main/install.ps1" -UseBasicParsing).Content
Invoke-Expression $install

# Mac / Linux:
bash <(curl -fsSL https://raw.githubusercontent.com/diegomondrik/sdad-v4/main/install.sh)

The installer checks Node 18+, Claude Code and ccstatusline; creates .claude/ with skills, agents and hooks (Windows); and initializes SPEC.md, LESSON_LIBRARY.md and .gitignore.

Before every session

npx ccstatusline@latest   # terminal 1 — context %, model, cost, branch
claude                    # terminal 2 — Claude Code

New project: the happy path

  1. $spec — you answer one question at a time (starting with project language). Claude proposes defaults; you only correct where it differs from your intent. You confirm the tier.
  2. $specout — you review the generated SPEC.md. This is the moment to argue: changing the Spec costs minutes, changing built code costs days. You approve.
  3. $build [feature] — Claude announces the increment (files, tests, docs, recommended model). If the 🧠 MODEL line does not match your session, you run /model + /effort. You approve and it builds.
  4. Automatic after each increment — real tests, $qa, a DECISIONS.md entry, §13 updated, and possibly a lesson candidate (you decide: yes / skip / edit).
  5. At session end — $pause compress generates the snapshot. Paste it at the start of the next session and Claude restores the full state without you explaining anything.

Existing project without SDAD

$docfinal runs 4 steps: a retroactive Spec inferred from the code (SPEC_RETROACTIVE.md — it never overwrites your SPEC.md), an authorship log, a full QA audit (report only, it fixes nothing without your OK), and up to 3 lesson candidates. From there the project follows the normal cycle.

Common beginner mistakes

MistakeDo this instead
Asking for code directly in chat, skipping $specSDAD will block it — by design. Invest 15 minutes in the Spec

9 · Context budget, lessons and hooks

Context budget

The context window is the scarce resource of a session. SDAD monitors it constantly: at 50% it warns (informational); at 65% it blocks new $build — you finish the current increment cleanly, run $pause compress, and start a new session. Sub-agents ($agent, $doc) run in isolated context: delegating heavy work to them is free for your session.

Lesson Library

After every $qa, if the increment revealed a transferable pattern (a bug likely to recur, an undocumented integration quirk, a pattern that simplified everything), Claude proposes one lesson. If nothing valuable emerged, silence — an entry is never forced. Lessons are tagged #stack / #phase and Phase 0 of every project surfaces the 2–3 relevant ones.

Hooks (Windows/PowerShell, active since v4.2)

HookWhenWhat it does
SessionStartOn session open / resumeRe-injects the COMPACT ANCHOR ([LOCK] decisions) + guarded git pull (fast-forward only)
PreCompactBefore context compactionWrites the anchor to disk so SessionStart can restore it

10 · FAQ

Can I use SDAD with any model?

Yes. The methodology is model-agnostic: the routing table speaks in tiers (FRONTIER/STANDARD/ECONOMY), not names. When a new model ships, only the tier mapping changes.

What if Claude finds a security issue in the middle of something else?

It flags it 🚨 regardless of phase. But it never fixes it alone: security and compliance findings always require your explicit approval.

Does SDAD need Pyplan's Analyst Agent?

No. SDAD is self-sufficient — it never depends on Pyplan's native agent.

Where do I see my project's state?

$pause: current phase, Spec status, tier, platform, context %, last increment with test results, open QA findings, and the recommended next step.

What is a [LOCK] decision?

A decision that is not reopened without your explicit override. They are the only ones carried in the COMPACT ANCHOR, so they survive context compaction and session changes.
