1 · Qué es SDAD y por qué existe

SDAD (Spec-Driven AI Development) es una metodología para desarrollar con Claude Code como herramienta principal. Su regla fundacional cabe en una línea:

🔒 Regla de oro Nunca se escribe código de producción antes de que el desarrollador apruebe una Spec. La única excepción es $docfinal, que documenta retroactivamente proyectos ya construidos.

¿Por qué? Porque el riesgo principal de desarrollar con IA no es que escriba mal el código — es que escriba rápido el código equivocado. Sin una especificación acordada, cada sesión improvisa decisiones, el alcance se mueve, y al tercer día nadie sabe por qué el sistema hace lo que hace. SDAD invierte el orden: primero acordamos qué y por qué (Spec), después el modelo ejecuta cómo (Build), y cada incremento cierra con control de calidad (QA) y registro de decisiones.

Todo queda escrito en archivos versionados en el repo — no en la memoria del chat:

ArchivoQué guarda
CLAUDE.mdLas reglas de la metodología (este es el "cerebro" que Claude Code lee en cada sesión)
SPEC.mdLa especificación aprobada: el contrato del proyecto
DECISIONS.mdCada decisión arquitectónica, con fecha, modelo y razón. Las marcadas [LOCK] no se reabren
LESSON_LIBRARY.mdLecciones transferibles entre proyectos (L-XX)
.claude/Skills especialistas, agentes aislados y hooks de sesión

2 · Las fases: el ciclo completo

0 · Contexto→ 1 · Requerimientos ($spec)→ 2 · Spec ($specout)→ 3 · Build ($build)→ 4 · QA ($qa)→ 5 · Docs ($doc)

Fase 0 — Análisis de contexto

Automática al abrir sesión. Claude lee el repo real (nunca pregunta lo que puede inferir), detecta si hay UI, recomienda un tier de compliance, y trae 2–3 lecciones relevantes de la Lesson Library. Si el proyecto estuvo >30 días sin $build, dispara $verify audit para revisar dependencias desactualizadas.

Fase 1 — Requerimientos guiados ($spec)

Claude hace una pregunta a la vez, siempre con un default propuesto — nunca un cuestionario. Recorre: alcance, flujos de usuario, modelo de datos, integraciones, reglas de negocio, performance, seguridad, tier de compliance y testing. La pregunta de tier es obligatoria, nunca se saltea.

Fase 2 — Generación de la Spec ($specout)

Genera el documento completo (§1 Visión … §13 AI Authorship Log) y lo escribe directamente en SPEC.md. Después pide tu aprobación. Sin aprobación, no hay $build.

Fase 3 — Desarrollo guiado ($build)

Trabaja en incrementos verticales: porciones completas de funcionalidad con sus tests. Antes de cada incremento, Claude anuncia qué archivos toca, qué tests va a correr, qué docs actualiza, y qué modelo/esfuerzo recomienda (ver routing). Después de escribir el código: corre los tests reales, ejecuta $qa, registra la decisión en DECISIONS.md y actualiza el §13 de la Spec.

Fase 4 — QA ($qa)

Cinco capas, en orden de prioridad: 🔐 Seguridad → 🏗️ Estructura → ⚡ Eficiencia → ✅ Best Practices → 🟠 Plataforma (solo Pyplan). Los hallazgos se numeran H-XX con severidad P0/P1/P2.

🔒 Gate de seguridad $qa automático nunca toca seguridad, compliance ni desviaciones de la Spec sin aprobación humana explícita. Los P0 se marcan 🚨 en cualquier fase.

Fase 5 — Documentación ($doc)

Se delega automáticamente a un sub-agente (no consume tu contexto) y escribe en /docs: README, referencia de API, documento de arquitectura, resumen de compliance.

5 · Routing de modelo y esfuerzo nuevo v4.3

No todas las fases necesitan el mismo modelo. Diseñar una Spec exige el mejor razonamiento disponible; ejecutar un incremento ya especificado no. v4.3 formaliza esto con una tabla de ruteo por fase usando tiers agnósticos de modelo — así la tabla sobrevive a cada release de modelos nuevos:

TierSignificaHoy, por ejemplo
FRONTIEREl mejor modelo de razonamiento disponiblefable, opus
STANDARDBalance costo/capacidadsonnet
ECONOMYEl más rápido y baratohaiku
FaseModelo · EsfuerzoPor qué
$spec / $specoutFRONTIER · highDecisiones abiertas, diseño de requerimientos
$build (por incremento)STANDARD · low — o FRONTIER · high si el incremento tiene riesgo o decisión abiertaEjecutar lo ya especificado es barato; decidir no
$qa incrementalSTANDARD · mediumRevisión acotada a un incremento
$qa full / $QA / $docfinalFRONTIER · highJuicio sobre el codebase completo
$verify / $docECONOMY–STANDARD · lowMecánico, delegable
$agent review/auditPinneado en frontmatter: opus · highVía B — el sub-agente trae su modelo
$agent testPinneado en frontmatter: sonnet · mediumGeneración guiada por contrato
$pause / $lesson / $flowEl modelo actual · lowNunca vale la pena cambiar para esto
⚠️ Importante: la sesión nunca cambia de modelo sola Claude Code no auto-conmuta modelo ni esfuerzo. El mecanismo es: al iniciar cada fase, Claude emite la línea 🧠 MODEL: [recomendado] · effort [nivel]; si tu sesión activa difiere, te lo señala y vos corrés /model y /effort. Solo $build bloquea la escritura de código ante un mismatch — las demás fases avisan una vez y siguen.

6 · SDAD para Pyplan

Cuando el proyecto declara PROJECT_PLATFORM: pyplan en el CLAUDE.md, SDAD se extiende con secciones, skills y checklists específicos de la plataforma. Nada del ciclo base cambia — se le agregan capas.

Secciones extra de la Spec (van antes de §1)

SecciónQué documenta¿Es gate?
§0 Platform ContextVersión de Pyplan, workspace, permisos, tipos de datos, convencionesNo
§A Data ArchitectureDiagnóstico del cliente, decisión de arquitectura, contrato de datos por fuenteSí — bloquea $build hasta aprobarse
§B Discovery LogSe inicializa vacío; registra los deltas de datos que aparecen durante $buildNo
§D MCP Tools CatalogSolo si el proyecto expone nodos @mcp_tool: nombre, descripción, parámetros tipados, retorno serializableSí — si existe, bloquea $build hasta aprobarse
💡 La lógica de los gates En proyectos Pyplan, el error más caro es construir el modelo sobre datos que no son como se asumió. §A obliga a acordar el contrato de datos antes de construir — igual que §9 Seguridad bloquea el build en proyectos Tier 3.

Deltas de datos: el protocolo de descubrimiento

Durante $build, la realidad de los datos del cliente suele desviarse de lo que dice §A. SDAD distingue dos casos:

Tipo de deltaEjemplosQué pasa
PequeñoError de formato, nulls, campo con otro nombre, volumen inesperadoSe resuelve en el nodo, se registra en §B, $build continúa
EstructuralEl dato no existe, granularidad incorrecta, la fuente es otra cosa🛑 $build se pausa. Se genera un reporte de gap para que el consultor lo lleve al cliente. Nunca se improvisa un workaround

Checklist de incremento Pyplan

Cada incremento Pyplan cierra verificando: todos los nodos nuevos tienen result= asignado y calculan sin error; sin dependencias circulares; las fuentes de los nodos de lectura coinciden con el contrato §A; los índices están sincronizados; los inputs tienen validaciones; los componentes visualizan datos calculados (no vacíos); y los deltas encontrados quedaron en §B.

MCP en Pyplan (nodos @mcp_tool)

Pyplan puede exponer nodos como herramientas MCP que otros clientes de IA invocan. SDAD trata esa superficie con disciplina extra porque es un contrato hacia afuera:

  • Docstring que explica la acción de negocio con precisión — un LLM externo tiene que poder invocarla bien solo leyéndola.
  • Todos los parámetros con Annotated[tipo, 'descripción'] — nada sin tipar.
  • Retorno serializable: nada de xarray crudo ni DataFrames sin .to_dict().
  • result = _fn (se asigna la función, no se la llama).
  • QA Layer 1 verifica: tokens OAuth nunca en logs ni resultados (P0), parámetros validados (P1), scope mínimo (P2).
⚠️ Build-via-AI El MCP de Pyplan permite que un cliente de IA modifique la aplicación directamente. SDAD lo permite con guardrails: cada modificación vía MCP es un incremento — requiere Spec aprobada, anuncio previo, tu aprobación, QA y registro en DECISIONS.md. El MCP no saltea el gate de la Spec. Además, el servidor MCP de Pyplan es v1: se documenta como dependencia externa en §7 y se recomienda fijar versión en §5.

7 · Cómo usarlo día a día

Instalación (una vez por máquina + una vez por proyecto)

# Windows (PowerShell) — desde el repo del proyecto:
$install = (Invoke-WebRequest -Uri "https://raw.githubusercontent.com/diegomondrik/sdad-v4/main/install.ps1" -UseBasicParsing).Content
Invoke-Expression $install

# Mac / Linux:
bash <(curl -fsSL https://raw.githubusercontent.com/diegomondrik/sdad-v4/main/install.sh)

El installer verifica Node 18+, Claude Code y ccstatusline; crea .claude/ con skills, agentes y hooks (Windows); e inicializa SPEC.md, LESSON_LIBRARY.md y .gitignore.

Antes de cada sesión

npx ccstatusline@latest   # terminal 1 — contexto %, modelo, costo, branch
claude                    # terminal 2 — Claude Code

Proyecto nuevo: el camino feliz

  1. $spec — contestás una pregunta por vez. Claude propone defaults; solo corregís donde difiere de tu intención. Confirmás el tier.
  2. $specout — revisás el SPEC.md generado. Este es el momento de discutir: cambiar la Spec cuesta minutos, cambiar el código construido cuesta días. Aprobás.
  3. $build [feature] — Claude anuncia el incremento (archivos, tests, docs, modelo recomendado). Si la línea 🧠 MODEL no coincide con tu sesión, corrés /model + /effort. Aprobás y construye.
  4. Automático tras cada incremento — tests reales, $qa, entrada en DECISIONS.md, §13 actualizado, y quizás un candidato a lección (decidís: yes / skip / edit).
  5. Al terminar la sesión — $pause compress genera el snapshot. Lo pegás al inicio de la próxima sesión y Claude restaura todo el estado sin que expliques nada.

Proyecto existente sin SDAD

$docfinal corre 4 pasos: Spec retroactiva inferida del código (SPEC_RETROACTIVE.md — nunca pisa tu SPEC.md), log de autoría, auditoría QA completa (solo reporta, no arregla nada sin tu OK), y hasta 3 candidatos a lección. Desde ahí el proyecto sigue el ciclo normal.

Errores comunes de novato

ErrorQué hacer en cambio
Pedir código directo en el chat salteando $specSDAD lo va a bloquear — y es a propósito. Invertí 15 minutos en la Spec
Aprobar la Spec sin leerlaLa Spec es el contrato: lo que no esté ahí, no existe para $build
Ignorar la línea 🧠 MODELUsar FRONTIER para todo quema cuota; usar ECONOMY para diseñar produce specs débiles
Seguir trabajando con contexto al 70%Al 65% SDAD bloquea nuevos $build: cerrá el incremento, $pause compress, sesión nueva

8 · Presupuesto de contexto, lecciones y hooks

Presupuesto de contexto

La ventana de contexto es el recurso escaso de una sesión. SDAD la monitorea siempre: al 50% avisa (informativo); al 65% bloquea nuevos $build — terminás el incremento en curso limpio, $pause compress, y sesión nueva. Los sub-agentes ($agent, $doc) corren en contexto aislado: delegarles trabajo pesado es gratis para tu sesión.

Lesson Library

Después de cada $qa, si el incremento reveló un patrón transferible (un bug que va a repetirse, un quirk de integración no documentado, un patrón que simplificó todo), Claude propone una lección. Si no hay nada valioso, silencio — nunca se fuerza una entrada. Las lecciones se etiquetan #stack / #phase y la Fase 0 de cada proyecto trae las 2–3 relevantes.

Hooks (Windows/PowerShell, activos desde v4.2)

HookCuándoQué hace
SessionStartAl abrir / reanudar sesiónRe-inyecta el COMPACT ANCHOR (decisiones [LOCK]) + git pull guardado (solo fast-forward)
PreCompactAntes de compactar contextoEscribe el anchor a disco para que SessionStart lo restaure
SessionEndAl cerrar sesiónAutocommit de docs whitelisteados (DECISIONS.md, LESSON_LIBRARY.md) — nunca código

Los equivalentes bash para Mac/Linux están pendientes (gap conocido de v4.3).

9 · Preguntas frecuentes

¿Puedo usar SDAD con cualquier modelo?

Sí. La metodología es agnóstica: la tabla de routing habla de tiers (FRONTIER/STANDARD/ECONOMY), no de nombres. Cuando salga un modelo nuevo, solo cambia qué modelo ocupa cada tier.

¿Qué pasa si Claude encuentra un problema de seguridad en medio de otra cosa?

Lo marca 🚨 sin importar la fase. Pero nunca lo arregla solo: los hallazgos de seguridad y compliance siempre requieren tu aprobación explícita.

¿SDAD necesita el Analyst Agent de Pyplan?

No. SDAD es autosuficiente — nunca depende del agente nativo de Pyplan.

¿Dónde veo el estado de mi proyecto?

$pause: fase actual, estado de la Spec, tier, plataforma, % de contexto, último incremento con resultado de tests, hallazgos QA abiertos, y el próximo paso recomendado.

¿Qué es una decisión [LOCK]?

Una decisión que no se reabre sin override explícito tuyo. Son las únicas que viajan en el COMPACT ANCHOR, así sobreviven compactaciones de contexto y cambios de sesión.
