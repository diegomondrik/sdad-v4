SDAD v6.0 — The Methodology

AI agents are capable.
The problem is discipline.

SDAD — Spec-Driven AI Development — is a methodology for using Claude Code as a disciplined engineering partner. It trades the illusion of speed for actual shipping confidence: structured requirements, coded gates, verifiable increments, and a lesson library that compounds across projects.

✗
"It built something different from what I asked." No requirements phase — Claude inferred the spec from the conversation.
✗
"It rewrote the whole codebase." No increment discipline — Claude over-engineered beyond the scope of the request.
✗
"Security issues appeared in production." No QA layer — code review was skipped to ship faster.
✗
"We lost context mid-project." No structured state — decisions lived only in the chat window.

Why SDAD exists

AI coding agents change what fails, not whether things fail

Claude Code can write production code in minutes. That speed makes the classical engineering discipline — requirements, spec, incremental delivery, QA — feel like overhead. It is not. The same problems that affect team software development affect AI-assisted development, but they surface differently and faster.

Without SDAD
  • Requirements live in the chat — lost when the context closes
  • One long increment: Claude writes everything, then the review catches problems too late
  • Decisions not recorded — team can't understand why choices were made
  • QA is manual or absent — security issues reach production
  • Each project restarts from zero — no learning accumulates
  • Context overflows: Claude loses track of the spec and improvises
  • Platform-specific invariants (Pyplan, Board) forgotten mid-build
  • Compliance tiers discovered in post-production audits
With SDAD
  • Requirements captured in SPEC.md — durable, versioned, reviewable
  • Small increments: announce → code → test → QA → commit atomically
  • DECISIONS.md records every architectural choice with rationale
  • 4-layer QA runs after every increment (Security is P0)
  • LESSON_LIBRARY.md captures transferable patterns across projects
  • $pause compress preserves state across context boundaries
  • Platform checklists run at every increment close
  • Compliance tier confirmed in Phase 1, gates activated immediately

The Harness Model

Hard gates live in code. Prompt rules are the fallback.

SDAD's governance axiom: rules that can be mechanically verified are enforced in code — a git pre-commit hook, a PreToolUse check, a check script that runs in CI. The methodology prompt fills the gap only where machine enforcement is not possible.

Design Principles

The decisions that define SDAD

Every constraint in SDAD exists because a real failure mode justifies it. These are not style choices — they are responses to observed breakdowns in AI-assisted development.

Principle 01
Spec Before Code
Claude is capable enough to build anything you describe. The risk is building the wrong thing completely. A spec that takes 30 minutes to produce prevents 3 days of rework.
Never write production code before SPEC.md is approved
Principle 02
One Increment at a Time
Each increment is announced, confirmed, coded, tested, QA'd, and committed atomically. An increment that silently expands scope becomes a regression. The announcement is the contract.
Announce → confirm → write → test → $qa → commit
Principle 03
Security is P0
Security findings are never auto-fixed. Every P0 or P1 finding from QA Layer 1 requires explicit developer approval before any change is applied. Speed does not override this.
$qa auto never touches security without human approval
Principle 04
Execute, Don't Simulate
Claude Code has direct filesystem and terminal access. SDAD uses it. Real files are read. Real tests are run. Results are actual — never simulated or described without execution.
Never simulate what you can execute
Principle 05
Lessons Compound
Each project generates knowledge. After every $qa run, SDAD evaluates whether the increment revealed a bug pattern, an integration quirk, or an architectural shortcut worth capturing. If so, it proposes one lesson candidate — the most valuable finding only. The developer approves, skips, or edits it, then it is written directly to LESSON_LIBRARY.md. The library is searchable by #stack and #phase tags. When a lesson describes something mechanically verifiable — a missing assignment, a circular dependency, a non-ASCII character — it is promoted from prose to a check in checks/. That check runs in CI and in git pre-commit, enforcing the lesson on every future project automatically. The pattern cannot recur silently.
$qa finding → lesson candidate → LESSON_LIBRARY → checks/ ratchet → CI enforcement
Principle 06
Evidence Before Judgment
The $audit command evaluates existing models without fabricating findings. If evidence cannot be acquired, the area is declared "not assessable" — a finding, not a silent skip. No hallucinated compliance.
Un-acquirable evidence = "not assessable", never an assumption

The Five-Phase Loop

Why each phase exists

The loop is not ceremony. Each phase has a specific failure mode it prevents.

01
$spec

Context Analysis & Requirements

Claude reads the repo before asking anything. Questions are one at a time, with proposed defaults. The developer is not writing a requirements doc — they are confirming or correcting Claude's inferences.

First question, always: project language — English or Spanish? The answer is locked as PROJECT_LANGUAGE in CLAUDE.md and governs every document, every interaction, and every generated output for the lifetime of the project. Code identifiers and comments stay in English regardless — only the developer-facing content switches. If the project language is not set, $spec asks it before anything else.

Prevents: building from chat-only requirements that disappear with the context window.
02
$specout

Spec Document

13 mandatory sections cover vision, flows, data model, architecture, business rules, security, definition of done, and an AI Authorship Log. Written to SPEC.md in the repo — versioned and reviewable.

Prevents: ambiguous build contracts, compliance surprises in production, scope disagreements mid-project.
03
$build [feature]

Guided Development

Each increment is announced before any code is written. The announcement names files, tests, and docs. Tests run after writing — not skipped. Every decision goes to DECISIONS.md.

Prevents: unscoped rewrites, untested code, decisions that exist only in someone's memory.
04
$qa

Quality Assurance

Four layers: Security (P0/P1/P2) · Structure · Efficiency · Best Practices. Platform Layer 5 activates for Pyplan and Board. Runs after every single increment — not at project end.

Prevents: security vulnerabilities reaching production, structural debt accumulating invisibly across increments.
05
$lesson / $pause

Learn & Pause

$lesson proposes the most valuable transferable finding from the QA run — not forced, not verbose. $pause compress packages session state for the next conversation. $eval verifies the methodology itself.

Prevents: knowledge loss, context boundary failures, methodology regressions shipped as updates.

Compliance

Tiers are detected, not assumed

SDAD reads the repo in Phase 0 and recommends a compliance tier. The developer confirms or overrides. The tier gates which sections are mandatory in SPEC.md, which skills auto-activate, and what the definition of done requires.

Tier 1

Standard

Internal / POC

Internal tools, productivity scripts, personal projects. No automatic skill activations. Standard definition of done.

  • Internal tools
  • POCs and prototypes
  • Personal projects
  • No user data
  • No external exposure
Tier 2

Business

Customer-facing / SaaS

Customer-facing products, SaaS applications, systems that handle user data. Auto-activates Compliance Reviewer.

  • Audit logging present
  • PII handling documented
  • Auth reviewed in §9
  • Sanitized error messages
Tier 3

Enterprise

Regulated / Corporate IT

Cloud deployments to corporate IT, healthcare, finance, government, ISO/SOC2. §9 must be complete before $build is allowed.

  • Threat model documented
  • Data flow diagram
  • Control matrix in SPEC.md
  • §9 build gate active
  • Regulation-specific skills

API key exposure, unprotected endpoints, PII in logs (P0) · missing input sanitization, weak auth (P1) · rate limiting, missing headers (P2). MCP tool scope and OAuth token handling when §D is active.

New in v6.0 — "Pyplan Audit Edition"

What v6 adds

v6 extends SDAD into a new lifecycle: auditing models that were not built with SDAD. It adds a five-dimension client-facing audit, business alignment without fabrication, and domain correctness profiles for finance and supply-chain.

$audit
Five-dimension Pyplan Audit
New command

A sibling to $docfinal: runs on an existing Pyplan model without a SPEC.md. Where $docfinal documents, $audit judges and recommends. The deliverable is a structured five-dimension client-facing report: Development, Security, Usability, Quality, Business.

Evidence is acquired per .sdad/audit/SCHEMA.md — Pyplan MCP read, .ppl export, manual artifacts. Areas where evidence cannot be acquired are declared "not assessable", never assumed. The report communicates intent-vs-delivered neutrally — evidence-based, relationship-aware, never accusatory.

Dimension 5a
Business Alignment
Evidence-first

Evaluates whether the model's logic aligns with the client's declared business objective. Requires owner elicitation — SDAD does not fabricate a business objective if none was provided by the owner. If no elicitation input exists, the alignment finding reads: "not assessable — no elicitation input."

This is a finding to surface, not a blank to fill. The methodology treats fabricated alignment as a liability risk — it could give false assurance that a broken model is strategically aligned.

Dimension 5b
Domain Correctness Profiles
domain-finance · domain-supply-chain

Domain-specific review profiles loaded on-demand when PROJECT_DOMAIN is set. Each profile carries rule-based checks: financial model profiles check for discount rate handling, NPV/IRR conventions, FX assumptions; supply-chain profiles check for lead-time treatment, demand aggregation patterns, and safety stock assumptions.

When a domain is detected but no profile exists, the finding reads: "not assessable — no domain profile." Confidence levels are always stated — an LLM domain profile raises the floor for review, it does not replace a client's Subject Matter Expert for high-stakes decisions.

$eval
Methodology Self-Evaluation
22 + 6 scenarios

The V-function of the harness model. Replays a golden dataset of scenarios that test SDAD's own rules — the spec-gate hook, the ASCII ratchet, $agent liveness, CLAUDE.md structural assertions, audit behavioral scenarios (evidence-first, not-assessable, domain gap). All 22 deterministic scenarios must pass before any tagged release.

$eval -Release also runs 6 LLM-smoke scenarios (non-deterministic) to catch behavioral regressions that static checks miss. If CLAUDE.md has changed since the last green $eval run, the SessionStart hook reminds the developer to re-run before the next client delivery.

Architectural Decisions

The "why" behind the methodology

SDAD is not a best-practices list assembled by committee. Each structural decision has a specific rationale. Many are locked — they survived long enough to prove their value and can't be reopened without a documented reason.

C-015 LOCK
"Why does $build block until the developer manually switches model, instead of auto-switching?"
Reproducibility and cost visibility. Auto-switching hides which model produced which code, breaking §13 AI Authorship Log accuracy and making it impossible to reproduce a result. The developer must consciously choose the model for each increment — this is intentional friction.
C-011 LOCK
"Why does the MCP-vs-CLI rule have a hard security gate?"
Security cannot be traded for token savings. A CLI wrapper chosen to save context budget must not add shell-injection risk, expose credentials in argv, or introduce fragile parsing. If it does, the vetted MCP stays — the cost reduction is overridden. This decision is non-negotiable.
C-004 LOCK
"Why does SDAD have no dependency on Anthropic's API — only on Claude Code?"
Team reach without infrastructure. A methodology that requires an API key and a server is a deployment. SDAD must run from a CLAUDE.md file — no server, no API endpoint. Any developer with Claude Code can adopt it in five minutes.
§2.0 LOCK
"Why is CLAUDE.md kept lean at ≤+60 lines per release?"
Context budget and reliability. Every line in CLAUDE.md consumes main-session context on every turn. Voluminous content goes to on-demand skills — loaded only when the trigger matches. The methodology prompt stays short; the skills stay deep.
L-01
"Why must all .ps1 and .sh files be pure ASCII?"
Fresh-machine installer reliability. Windows PowerShell 5.1 reads UTF-8 without BOM incorrectly, and POSIX shells on minimal installations may misread multi-byte sequences. Non-ASCII bytes in installer scripts cause silent failures on the machines where reliability matters most.
I-8
"Why does $audit not reuse $docfinal internals?"
Different intent, different output. $docfinal generates documentation for a team. $audit generates a client-facing judgment report for a consultant. The evidence contract, the five-dimension structure, and the neutrality requirements are different enough to warrant a separate lifecycle. Shared scaffolding (AUDIT_ACTIVE sentinel, spec-gate allowlist) without shared command logic.

Fit & Misfit

When SDAD fits — and when it does not

SDAD adds structure. Structure takes time. The question is whether the structure saves more time than it costs — and it almost always does when the project runs beyond a single session.

Use SDAD for
  • Projects that will run more than one Claude Code session
  • Projects where requirements are not perfectly clear at the start
  • Any feature that touches user data, authentication, or external APIs
  • Client-facing work where a consultant must document decisions
  • Pyplan or Board models with complex data architectures
  • Audit of an existing Pyplan model before client delivery
  • Teams where multiple developers will read the code
  • Projects regulated or subject to compliance review
  • Any project where "what Claude built" must be reproduced later
Consider skipping for
  • One-shot scripts you'll run once and discard
  • Pure research or exploration sessions (no production code)
  • Prototypes where requirements change every 20 minutes
  • Projects where the developer already has a complete implementation plan
Even then —

Even for quick scripts, running $docfinal afterward takes 5 minutes and produces a spec, QA report, and authorship log from the code as-written. It's the minimum viable SDAD workflow.
