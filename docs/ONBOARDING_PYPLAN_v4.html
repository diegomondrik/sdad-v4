Activating the Pyplan layer

SDAD activates Pyplan-specific behavior by reading one line in SPEC.md: 

PROJECT_PLATFORM: pyplan

This declaration must appear in §0 — Platform Declaration at the top of SPEC.md. Once present, four Pyplan skills load automatically and additional sections are added to the Spec structure.

Declare the platform at project start — not mid-build If PROJECT_PLATFORM: pyplan is missing from §0, Pyplan skills do not activate, §A gate checks are skipped, and data delta handling rules are not enforced. Always set it before running $spec.

What activates automatically

ActivationWhat changes
Pyplan skills (5) pyplan-diagram, pyplan-interfaces, pyplan-qa-platform, pyplan-spec-context, pyplan-mcp load on-demand by trigger
SPEC additions §0, §A (Data Architecture), §B (Discovery Log), and §D (MCP Tools Catalog — conditional) prepended to the standard 13-section structure
$build gates §A must be approved before any $build increment. §D must be approved when project declares @mcp_tool nodes.
$qa Layer 7 Pyplan Platform checks (PP-XX findings) appended to the standard QA run
$build checklist Pyplan-specific increment close checklist runs before any increment is marked complete

Pyplan SPEC structure

A Pyplan SPEC.md has three additional sections at the front (§0, §A, §B), followed by the standard 13 sections. §C (Brand & Visual Identity) is optional — include it when the project has a client-facing UI with brand requirements.

§0
Platform Declaration
Declares PROJECT_PLATFORM: pyplan and locks Pyplan behavior for the session
Required fields: PROJECT_PLATFORM, PYPLAN_VERSION, DEPLOYMENT, CLIENT, COMPLIANCE_TIER. This section never changes after initial approval.
§A
Build Gate
Checklist — all items must be checked before $build is allowed
Contains a checklist of conditions: §0 approved, §1 Vision agreed with client, roles defined, at least one functional flow documented, data sources identified and accessible, compliance tier locked, no blocking Open Decisions. Any unchecked item blocks $build. Sealed after first build approval.
§B
Living Model State
Updated automatically at every increment close — the fastest way to resume a session
Tracks modules (complete / in-progress / not started), data connections (connected / pending / blocked), open issues, and the next planned increment. Claude reads §B first at session start. If §B conflicts with §13, §B takes precedence for current state — §13 is the historical record.
§C
Brand & Visual Identity (optional)
Required only when the project has a client-facing UI with brand guidelines
Populated during $spec when the project has a visible interface for a named client. Activates the Brand Design skill automatically. Records brand material source, primary color, primary font, Brand Token Sheet status, and client approval date. Omit entirely for backend-only work or internal tools.
§D
MCP Tools Catalog (conditional)
Required only when the project exposes at least one @mcp_tool node. Gate section: blocks $build until approved.
Documents each MCP tool: node identifier, tool name, description (written for an external LLM), parameter names + types + Annotated descriptions, return type, serialization notes. If the project has no @mcp_tool nodes, omit §D entirely. The pyplan-mcp skill activates automatically when §D is detected or @mcp_tool keywords appear.
§1–§13
Standard SDAD sections
Vision, Users, Flows, Data Model, Architecture, Business Rules, Integrations, Testing, Security, DoD, Out of Scope, Open Decisions, AI Authorship Log
§5 (Technical Architecture) and §7 (Integrations & Data Sources) have Pyplan-specific required additions: module structure, dimension catalog, driver nodes, and a structured data source table with refresh cadence and owner.
$build is blocked until §A is fully checked When $build is called, Claude reads §A first and reports status. Any unchecked item triggers a block with a list of what needs to be resolved. No exceptions.

Two surfaces

Every Pyplan application has two completely separate surfaces. Each has its own skill and its own set of conventions. Understanding this split is the most important mental model for Pyplan development.

Code Surface — Influence Diagram
  • Nodes with Python logic
  • Dependency graph (arrows = data flow)
  • Data Reading, Variable, Input, Index nodes
  • All transformations and calculations
  • No visual output — pure data pipeline
Skill: pyplan-diagram
Interface Surface — Dashboards
  • Components placed in a grid (Tables, Charts, KPIs)
  • Reads from diagram nodes
  • Index components write selections back to diagram
  • Input components write user values back to Input nodes
  • Everything the end user sees and touches
Skill: pyplan-interfaces

These two surfaces never overlap. When a task involves node code or data transformations, use pyplan-diagram. When a task involves components, filters, layouts, or brand application, use pyplan-interfaces.

Node types

Every node in the influence diagram has a type. The type determines what it can store and how it connects to the interface.

Variable
Generic Python logic. Most common type. Returns any Python object.
Data Reading
Connects to CSV, DB, Excel, API. Returns DataFrame or xarray.
Index
Stores a dimension list (Product, Region, Year). Used for index sync.
Input — Scalar
Single user-editable value (rate, flag, threshold). Always needs validation rule.
Input — Form
Tabular editable data stored in DB. Used for budget entry tables.
Input — Cube
Multidimensional editable data (Product × Region × Month planning matrices).
Report
Combines multiple nodes into a structured output. Used for complex dashboard outputs.
Module
Container grouping related nodes hierarchically. Reduces diagram clutter.

The result= convention

Every codable node must assign its output to result as the final line. Pyplan uses this value as the node's output for downstream nodes and interface components. Forgetting result = is the most common Pyplan error — the node evaluates but returns None, and downstream nodes fail silently. 

# All local variables use _ prefix
_df = pd.read_csv("data/sales.csv")
_df["date"] = pd.to_datetime(_df["date"])
_df = _df[_df["year"] == 2025]

result = _df   # always the last meaningful line

Recommended module structure

App root
├── 00_connections    ← DB connections, file paths, credentials
├── 01_inputs         ← all Input nodes (scalar, selector, form, cube)
├── 02_indexes        ← all Index nodes
├── 03_data           ← Data Reading nodes and raw processing
├── 04_calculations   ← business logic transformations
├── 05_outputs        ← Report nodes and outputs for interfaces
└── 99_utilities      ← helper functions, constants, shared logic

No node in 04_calculations should connect directly to a DB or file — that belongs in 03_data. This separation makes debugging and data source changes significantly easier.

SDAD does not depend on Pyplan's native Analyst Agent The Analyst Agent is considered immature. All methodology features — QA, spec navigation, data delta detection — work standalone without it.

Build flow

The $build flow for Pyplan projects is identical to the standard SDAD flow, with two additions: the §A gate check at the start, and a Pyplan-specific increment checklist at the close.

Before each increment

Claude announces the increment and waits for approval:

🔨 INCREMENT [N]: [feature name]
Files: [nodes and modules to create or modify]
Tests: [how the increment will be verified]
Docs: [SPEC.md sections to update]
Dependencies: [what must be done first]
──────────────────────────────────────────────────────
[Waiting for approval]

Pyplan increment close checklist

Before marking any increment complete, Claude runs this checklist:

  • Diagram: All new nodes have result = assigned and calculate without error
  • Diagram: No circular dependencies introduced
  • Diagram: Data source of Data Reading nodes matches §A data contract
  • Interface: All new indexes are synchronized
  • Interface: Inputs have type and range validations configured
  • Interface: Visualization components display calculated data (not empty)
  • Discovery: Any data deltas found during this increment recorded in §B and DECISIONS.md
  • Discovery: If structural delta found — $build paused, data gap report generated
  • MCP (when §D present): Each new @mcp_tool node has a precise docstring explaining the business action
  • MCP (when §D present): All parameters use Annotated[type, 'description'] — no untyped parameters
  • MCP (when §D present): Return value is JSON-serializable (no raw xarray, no bare DataFrames)
  • MCP (when §D present): result = _fn assigned — not result = _fn()
  • MCP (when §D present): §D entry created or updated for this tool

After each increment

  1. Run tests — report actual result
  2. Run $qa (includes Pyplan Layer 7)
  3. Write DECISIONS.md entry and HUB BLOCK
  4. Update SPEC.md §13 (AI Authorship Log)
  5. Update §B (Living Model State) — modules, data connections, open issues, next increment

Data delta handling

During $build, the actual data sources sometimes differ from what was specified in §A. The response depends on the severity of the difference.

Small delta — $build continues
  • Format error, nulls, unexpected volume
  • Wrong field name (but data exists)
  • Resolve in the node
  • Record in §B: assumption, finding, resolution, impact
  • Note in §13: "Delta: [description]"
  • Continue to next step
Structural delta — $build pauses
  • Data does not exist at all
  • Wrong granularity
  • Source completely different than spec'd
  • $build paused immediately
  • Data gap report written to §12
  • Consultant takes report to client
  • $build resumes only when resolution is recorded in §B
Never improvise a workaround for a structural delta If a data source does not exist or has the wrong granularity, $build stops. The gap report goes to the consultant, who brings it to the client. Only an approved resolution recorded in §B unlocks the build again.

MCP tools (since v4.1)

Pyplan MCP (v1) allows AI clients to connect to a running Pyplan instance via OAuth 2.1, discover application-specific tools, and execute them with structured parameters. In SDAD, MCP tools are first-class deliverables with their own gate section (§D), dedicated skill (pyplan-mcp), and QA checks.

Pyplan MCP is a v1 server First release as of 2026. Treat as an external dependency whose API may change across Pyplan updates. Document in §7 and flag in $verify. Do not assume stability.

The @mcp_tool pattern

For a node to be discoverable as an MCP tool, its result must be a decorated function:

from pyplan_core.classes.ai.Agent import mcp_tool
from typing import Annotated

@mcp_tool
def _fn(
    amount: Annotated[float, 'Amount to convert'],
    source_currency: Annotated[str, 'Source currency code, e.g. USD'],
    target_currency: Annotated[str, 'Target currency code, e.g. EUR'],
) -> dict:
    """
    Convert an amount from one currency to another using the configured rate source.
    Returns: source_currency, target_currency, original_amount, converted_amount, rate.
    """
    rate = exchange_rates.get_rate(source_currency, target_currency)
    return {
        'source_currency': source_currency,
        'target_currency': target_currency,
        'original_amount': amount,
        'converted_amount': amount * rate,
        'rate': rate,
    }

result = _fn   # assign the function — do not call it
RuleCorrectWrong
Assignment result = _fn result = _fn()
Parameters Annotated[float, 'description'] amount: float (no description)
Return Plain dict, list, or scalar Raw DataFrame or xarray object
Docstring Precise — explains what the tool does for an external LLM Vague ("converts data")
State Uses only declared parameters Reads from mutable session state

Build-via-AI guardrails

Pyplan MCP includes a capability to build and modify application logic and interfaces directly in a running Pyplan instance using natural language. SDAD allows this capability with the same discipline as $build:

  1. Spec must be approved before any build/modify action — MCP does not bypass the Spec gate
  2. Each AI-driven modification is announced as an increment and requires developer approval
  3. After execution: DECISIONS.md entry written + §13 AI Authorship Log updated
  4. $qa runs on the modified increment — no increment is complete without QA
  5. MCP surface checklist runs for any @mcp_tool node touched
No Spec, no build — even via MCP If $spec has not been run and approved, Claude blocks any Pyplan MCP build/modify action and redirects to $spec. The Spec gate applies regardless of how the modification is being made.

QA — Layer 7 (Pyplan Platform)

When $qa runs on a Pyplan project, it adds a 7th layer to the standard 6-layer SDAD QA. Findings from this layer are numbered PP-01, PP-02… independently from the base H-XX findings.

Layer 7 sectionWhat is checked
7.1 Node Design Single responsibility per node, naming conventions, no circular dependencies, no dead nodes, reusable nodes factored into utilities
7.2 Data Flow Data sources isolated from transformations, currency/date assumptions explicit, dimension consistency, no silent NaN propagation
7.3 Interface & Presentation No raw data shown to end users, navigation follows §3 functional flows, selector nodes grouped, edge cases handled visually
7.4 Performance Heavy nodes not downstream of volatile inputs, precomputable nodes separated, no vectorizable loops, no repeated file imports
7.5 Delivery Conventions §A gate conditions met, client-visible text in correct language, no debug scaffolding, exports tested with real data, permissions match client roles
7.6 MCP Tools (when §D present) All nodes in §D have @mcp_tool and result = _fn. All parameters use Annotated[...]. Docstrings precise enough for an external LLM. Returns JSON-serializable. OAuth token not exposed in results (P0). No path to arbitrary code execution via parameters (P1).

PP finding format

PP-01 🔴 [7.1 Node Design] — Node sales_by_region has multiple responsibilities
What: The node both filters raw data and calculates variance in a single block.
Where: Module 04_calculations → sales_by_region
Fix: Split into two nodes: sales_filtered (filter) and sales_variance (calculation)

Delivery status

After all PP findings, the QA report emits a combined summary with delivery status. Delivery is BLOCKED if any 🔴 finding is unresolved or the §A gate has unchecked items. Delivery is READY only when both conditions are clear.

Commands reference — Pyplan

CommandPyplan behavior
$spec Runs §0 (platform context) first, then asks Pyplan-specific questions about modules, data sources, user roles, and client conventions before the standard SDAD questions.
$specout Generates the full Pyplan SPEC: §0, §A (Build Gate), §B (Living Model State), then §1–§13 with Pyplan additions to §5 and §7.
$build [feature] Blocked if §A is not approved. Blocked if §D is present and not approved. Runs Pyplan increment checklist (including MCP surface) at close. Updates §B automatically.
$qa Runs standard 6-layer QA plus Layer 7 (PP-XX findings). Emits combined delivery status.
$pause Shows §A gate status, §B summary (modules complete/in-progress), open PP findings, and data connections status.
$skills brand-design Activates Brand Design skill to produce the Brand Token Sheet. Required before building any client-facing interface. Auto-activates when §C is populated.
Quick start sequence for a new Pyplan project
  1. Run project initializer: bash <(curl …/project-init.sh)
  2. Open SPEC.md and declare PROJECT_PLATFORM: pyplan in §0
  3. Start Claude Code: claude
  4. Run $spec — Claude reads the repo and starts guided requirements
  5. Run $specout — generates full Pyplan SPEC with §0/§A/§B
  6. Review and approve §A gate — every item must be checked
  7. Run $build — first increment unlocked
