G7 AI Development Methodology

SDAD v4.3 — Developer Guide

Complete methodology reference for developers using Claude Code with SDAD.

What is SDAD?

SDAD (Spec-Driven AI Development) is G7's development methodology for teams using Claude Code as their primary AI development tool. It structures AI-assisted development into five sequential phases, enforces spec-first discipline, and provides a shared quality framework across all project types.

SDAD solves a specific problem with AI-assisted development: without structure, Claude Code tends to write code immediately, make implicit decisions that conflict with later requirements, and produce outputs that diverge from what the client actually needs. SDAD prevents this by requiring an approved Spec before any code is written.

Core principle

Never write production code before the developer approves a Spec.

Every project follows: Context Analysis → Requirements → Spec → Build → QA. Claude Code has direct filesystem and terminal access — it reads actual files, runs actual tests, and writes directly to the repo. It never simulates what it can execute.

Session setup

Configure the status bar once per machine, then just start Claude Code:

# Once per machine — interactive TUI: enable Model, Thinking Effort, Context %, Session Cost, Git Branch
npx ccstatusline@latest

# Every session — Claude Code (the bar renders at the bottom automatically)
claude

ccstatusline writes the statusLine entry into ~/.claude/settings.json and shows the current model, thinking effort, context usage percentage, session cost, and git branch in real time inside Claude Code. It is the primary indicator for context budget warnings at 50% and 65%. Re-run the command to change widgets.

Phase overview

Phase 0

Context Analysis — automatic

Claude reads the repo, detects platform, detects UI presence, and recommends a compliance tier. No command needed — runs at session start.

Phase 1 — $spec

Requirements

Guided discovery — one question at a time with a proposed default. Covers scope, user flows, data model, integrations, business rules, performance, security, and compliance tier.

Phase 2 — $specout

Spec Document

Generates the full 13-section SPEC.md and writes it to the repo root. Requires developer approval before $build is allowed.

Phase 3 — $build

Guided Development

Vertical increments with tests. Each increment announced before coding, tests run after writing, QA and decision log updated automatically.

Phase 4 — $qa

Quality Assurance

Layered quality review across Security, Structure, Efficiency, and Best Practices. Pyplan projects add a 5th Platform layer. Findings numbered H-01, H-02…

Phase 0 — Context Analysis

Phase 0 runs automatically at the start of every session. Claude reads the repo and produces a Context Analysis block covering:

  • Detected tech stack and key dependencies
  • Platform declaration (Pyplan, generic, etc.) if present in SPEC.md §0
  • UI presence — if detected, the Frontend skill is suggested
  • Compliance tier recommendation based on project context
  • Architecture flags (⚠️ ARCH:) from the AI Architect skill
  • Tooling gaps (⚠️ TOOLING:) from the AI Engineer skill
Phase 0 never blocks Flags surfaced in Phase 0 are informational. They do not prevent Phase 1 from starting. Security and compliance flags are still surfaced — the developer decides what to act on.

Phase 1 — Requirements ($spec)

$spec runs guided requirements discovery. Claude reads existing files first, then asks one question at a time with a proposed default — the developer confirms or overrides.

Standard question sequence:

  1. Project scope and objective
  2. User flows and main interactions
  3. Data model and key entities
  4. Integrations and external services
  5. Business rules and constraints
  6. Performance and scalability requirements
  7. Security and credentials handling
  8. Brand and visual identity (when UI is detected — see §C)
  9. Compliance tier — always asked, never skipped

The compliance tier question

Claude always asks:

"What's the deployment context?
 (1) Internal tool / POC — Tier 1 Standard
 (2) Customer-facing product / SaaS — Tier 2 Business
 (3) Regulated environment / corporate IT / cloud enterprise — Tier 3 Enterprise
 Based on what I see in this repo, I recommend: [Tier N — reason]"

The tier is locked on confirmation. Tier-specific skills and Definition of Done additions activate immediately.

When all areas are covered, Claude suggests running $specout.

Phase 2 — Spec Document ($specout)

$specout generates the full Spec document from Phase 1 requirements and writes it directly to SPEC.md in the repo root. No manual paste step.

For Tier 2/3 projects, §9 (Security & Compliance) is mandatory and must be complete before the Spec is approved. For Tier 3 and Pyplan projects, the respective gate sections (§9 / §A) block $build until approved.

Developer approval required before $build After $specout, Claude asks for explicit approval. $build is blocked until the Spec is approved. This is the primary quality gate in SDAD.

Phase 3 — Build ($build)

$build works in vertical increments — each increment is a self-contained piece of working functionality with tests. Before writing any code, Claude announces the increment and waits for approval.

Increment announcement format

🔨 INCREMENT [N]: [feature name]
Files: [list of files to create or modify]
Tests: [unit / integration / E2E — will be executed after writing]
Docs: [README update / API doc / inline comments required]
Dependencies: [what must be done first]
──────────────────────────────────────────────────────
[Wait for developer approval, then write code, then run tests immediately]

After each increment

  1. Run the project's test command — report actual result (pass count, failures, errors)
  2. Run $qa on the increment
  3. Write DECISIONS.md entry and HUB BLOCK
  4. Update SPEC.md §13 (AI Authorship Log)

$build gates

ConditionEffect on $build
SPEC.md not found Blocked. Claude reads the repo and offers $spec or $docfinal.
Spec not yet approved Blocked until developer approves SPEC.md.
Context budget at 65% Blocked after current increment. Finish cleanly, then run $pause compress.
Tier 3 — §9 incomplete Blocked until §9 Security & Compliance is complete and approved.
Pyplan — §A unchecked items Blocked until all §A Build Gate items are checked.
No test command found Flagged before writing code. Developer must define the test command.

Phase 4 — QA ($qa)

$qa runs a layered quality review. Findings are numbered H-01, H-02… and classified as: must fix, should improve, or style suggestion.

QA modes:

CommandScope
$qaIncremental — last $build increment only (auto mode)
$qa reviewManual — per-finding approval before any fix is applied
$qa fullFull project audit across all files
$QAStandalone audit — SDAD-Aware or Standalone mode

QA layers

Layers run in priority order. Security findings (P0) always surface first.

🔐

Layer 1 — Security

P0: API key exposure, unprotected endpoints, PII in logs. P1: missing input sanitization, weak auth. P2: rate limiting, missing security headers.

🏗️

Layer 2 — Structure

Architecture consistency, separation of concerns, error handling, context flow, tight coupling between components.

⚡

Layer 3 — Efficiency

Token usage, redundant LLM calls, conversation history management, unbounded loops, latency bottlenecks.

✅

Layer 4 — Best Practices

Readability, maintainability, code duplication, naming conventions, documentation gaps.

🟠

Layer 5 — Platform (Pyplan only)

Nodes missing result=, unsynchronized indexes, inputs without validations, circular dependencies, interface conventions. Findings numbered PP-01, PP-02… When §D is present: additionally audits @mcp_tool nodes for decorator + assignment correctness, Annotated parameters, docstring precision, serializable returns, and agent independence. MCP security checks (OAuth token exposure P0, parameter injection P1) run in Layer 1.

$qa auto never applies security or compliance fixes without approval Any finding in Layer 1 (Security) or compliance-related findings require explicit developer approval before any change is made. This applies in all QA modes.

Compliance tiers

Tier is detected in Phase 0 and confirmed in Phase 1. Once confirmed, it is locked for the project and activates additional skills and DoD requirements.

Context budget

Claude monitors context usage throughout the session and emits warnings at two thresholds. These are the only times context warnings appear — never in between.

0% 50% — soft warning 65% — hard warning 100%
ThresholdWarningAction required
50% — Soft ⚠️ Extended session. Consider starting fresh after current increment. Informational. Continue normally.
65% — Hard 🔴 Blocking $build after current increment. Finish current increment cleanly (tests + QA). Then run $pause compress, save snapshot, start new session.

Sub-agents run in isolated context — their usage does not consume the main session budget. This is why expensive operations like $doc and $agent review are always delegated.

Sub-agent delegation

SDAD delegates automatically to sub-agents when all three conditions are met: (1) the task operates on files already committed to the filesystem, (2) the task does not require knowledge of decisions made in this session, and (3) the task is expensive in context.

Always delegatedNever delegated
$doc (all variants) · $agent review · $agent test · $agent audit $qa after $build · $spec / $specout · $build

Agent definitions live in .claude/agents/:

  • code-reviewer.md — architectural review
  • test-generator.md — test suite generation
  • security-auditor.md — security audit
Sub-agent mechanics are silent Delegation is automatic. You see only the final result — not the agent execution. If the agent output file (.sdad/agent_output.tmp) is empty or missing, an error is surfaced before Claude proceeds.

HUB BLOCK — decisions log

After every completed $build increment, Claude generates a HUB BLOCK and writes it directly to DECISIONS.md in the repo root. It captures the key architectural or implementation decision made in that increment. 

════════════════════════════════════════════════════════
📋 HUB BLOCK — DECISIONS_[PROJECT].md
════════════════════════════════════════════════════════
Date:                   [YYYY-MM-DD]
Increment:              [N] — [feature name]
Model:                  [model used]
Decision:               [one-line summary]
Rationale:              [one-line rationale]
Alternatives:           [brief — or "none"]
Impact:                 [files changed, dependencies added, patterns introduced]
════════════════════════════════════════════════════════

The HUB BLOCK is also shown in-session after each increment for the developer to copy to the project hub file (hub/DECISIONS_[PROJECT].md) if using a shared hub.

Lesson Library

After every $qa run, Claude evaluates whether the increment revealed a pattern worth capturing. If triggered, one candidate is proposed. The developer approves, skips, or edits it. Approved entries are written directly to LESSON_LIBRARY.md.

Lesson capture triggers:

  • A bug or failure pattern likely to recur in other projects
  • An integration quirk not documented in official documentation
  • An architectural or prompt pattern that significantly simplified the solution
  • A Pyplan-specific pattern (node design, interface structure, data handling)

If nothing is lesson-worthy, capture is silent — no proposal is made.

Session management

$pause — session state

$pause shows a full session state snapshot:

  • Current Phase and Spec status per section
  • Compliance tier and platform
  • Context budget %
  • Last increment + test result
  • Open QA findings (H-XX) and PP findings (Pyplan)
  • Active skills
  • Decisions log count and last entry
  • Flows defined
  • Next step recommendation

$pause compress — cross-session continuity

$pause compress generates a compact Session Snapshot. Paste it as the first message in the next session — Claude restores all state without any re-explanation.

The snapshot includes: phase, spec status per section, compliance tier, platform, completed increments summary, open QA findings, open decisions, AI Authorship Log summary, Lesson Library summary, active skills, context budget %, flows defined, and exact next step.

Always run $pause compress before ending a session Restoring state from a snapshot at the next session start takes seconds and ensures no context is lost between sessions.

Brand design workflow

When a project has a client-facing UI for a named organization, the Brand Design skill produces a Brand Token Sheet — a structured document that translates brand identity into implementation-ready design tokens. It is the single source of truth for all visual decisions.

Activation

During $spec, when a UI is detected and the client is a named organization, Claude asks whether brand guidelines exist. The answer is recorded in SPEC.md §C. The Brand Design skill activates automatically when §C is populated.

Three phases

  1. Phase 1 — Brand Discovery: Classify available material (full brandbook, partial, logo only, website only, or nothing). Extract color tokens, typography, logo rules, spacing, and shape. Flag gaps.
  2. Phase 2 — Brand Token Sheet: Produce the structured deliverable with all CSS custom properties. Mark each token as [explicit], [inferred], or [proposed]. Requires client approval.
  3. Phase 3 — Application Guidance: Hand off approved tokens to the Frontend skill or Pyplan Interfaces skill. These skills consume the tokens — they do not make independent color or typography decisions.
Once the Brand Token Sheet is approved, any deviation is a finding BD-[N] findings are raised in QA for any hardcoded color or font that bypasses the token system. Classified as 🚨 Must fix (logo misuse, wrong color family) down to 💡 Suggestion.

Pyplan projects

Pyplan projects use a superset of the standard SDAD flow with platform-specific additions. Declare PROJECT_PLATFORM: pyplan in SPEC.md §0 to activate the full Pyplan layer.

Key differences from standard projects:

  • Additional Spec sections: §0 (Platform), §A (Data Architecture), §B (Discovery Log), §D (MCP Tools Catalog — conditional)
  • $build blocked until §A is approved. Also blocked if §D is present and not approved.
  • Pyplan increment close checklist runs before any increment is marked complete (includes MCP surface when §D present)
  • Data delta handling rules: small deltas resolved inline, structural deltas pause $build
  • $qa adds Layer 5 with PP-XX findings. When §D present: also audits @mcp_tool nodes and adds MCP security checks to Layer 1.
  • MCP tools (since v4.1): Pyplan MCP server (v1) allows AI clients to discover and execute @mcp_tool nodes. Build-via-AI capability allowed with same Spec-gate discipline as $build.
See the full Pyplan guide docs/ONBOARDING_PYPLAN_v4.html covers the Pyplan layer in depth: the two surfaces, node types, the result= convention, data delta handling, MCP tools, and the QA Layer 7 format.

Behavior rules

These rules govern Claude Code's behavior in every SDAD session. They override default behavior and are enforced unconditionally.

RuleWhy
Read actual files before asking questions Never ask what can be inferred from the repo
Run actual tests after every $build increment Never simulate test execution
Write SPEC.md to the repo on $specout Spec must exist as a file — not just in the chat
Ask the compliance tier question in Phase 1 Never skip — tier affects DoD, skills, and gates
Announce increments before coding Developer must approve what will be built
Include docs update in every increment announcement Documentation is part of the DoD, not optional
Mark critical security issues with 🚨 regardless of phase Security findings are never deferred
$qa auto never applies security or compliance fixes without approval These changes require explicit developer confirmation
Update SPEC.md §13 after every increment AI Authorship Log must be current
Write DECISIONS.md and HUB BLOCK after each increment Decision record enables cross-session continuity
Lesson capture is silent when nothing is worth capturing Never force an entry — quality over completeness
$agent delegation is automatic Never ask the developer which tasks to delegate
$verify runs automatically when $build adds a new dependency Dependency documentation must be current before coding
