¿Qué agrega SDAD al modelo del video?

El video describe 4 componentes (Contexto, Herramientas, Memoria, Validar). SDAD los implementa todos y agrega dos más: el Spec Gate (ningún código sin contrato aprobado) y el Modelo/Esfuerzo Routing (qué modelo usar en cada fase, anunciado antes de cada increment). El result es un arnés que no solo controla cómo trabaja la IA, sino también si tiene permiso de trabajar.

Gates: el núcleo de la disciplina

Código (hook)

El PreToolUse hook rechaza Write/Edit sobre archivos de código cuando SPEC.md no existe o no está aprobado. Imposible violar — es estructura, no prompt.

Aprobación humana

La transición Phase 2→3 requiere que el developer confirme el SPEC. Security findings en QA requieren aprobación antes de cualquier fix.

Automático

Phase 0 corre siempre al iniciar sesión. $qa corre siempre después de $build. Sub-agentes se delegan automáticamente para $doc, $agent review/test.

Los 3 pilares — Video vs SDAD

El video define tres pilares del harness engineering. SDAD los implementa todos, con mayor formalidad en cada uno.

📁
El repo como sistema
Video: Todo el arnés vive en ficheros del repo — agents.md, featurelist.json, progress/. El modelo lee ficheros para saber qué hacer.
SDAD v5
SPEC.md como contrato de 13 secciones.
DECISIONS.md con HUB BLOCK por increment.
LESSON_LIBRARY.md para conocimiento cross-proyecto.
§13 AI Authorship Log — attributabilidad por modelo/fecha.

Más estructurado que featurelist.json; más auditable. La Spec es un contrato, no solo una lista.
🤖
Orquestación multiagente
Video: Líder + subagentes con contexto aislado. Subagentes escriben en progress/ para evitar "el teléfono descompuesto".
SDAD v5
Sub-agentes con contexto aislado para $doc, $agent review/test/audit.
.sdad/lib/agent-run: timeout 600s + guard de output vacío.
HANDOFF BLOCK: resultado estructurado por sub-agente.
Routing modelo/esfuerzo: FRONTIER/STANDARD/ECONOMY por fase.

SDAD agrega detección de agentes zombie y routing explícito de modelo. El video no los menciona.
✅
Verificación
Video: La IA debe demostrar que algo funciona (tests, browser), no solo declararlo. El reviewer puede auto-mejorarse modificando su .md.
SDAD v5
$qa automático después de cada $build — 4 capas.
Tests reales — never simulate. Pass count, failures, errores reportados.
$eval golden dataset — valida la metodología misma.
Auto-mejora gobernada: el agente propone updates al CLAUDE.md del proyecto (paso 5.5), el developer aprueba.

Diferencia de filosofía: el video permite auto-modificación autónoma. SDAD mantiene al developer en el loop.

Frase de síntesis

"SDAD impone un contrato antes de que empiece el código, registra cada decisión mientras se construye, y verifica cada resultado cuando termina — todo dentro del repositorio, todo atribuible a un modelo y un momento."

El Axioma de Gobernanza

Restricciones estructurales en código > reglas en el prompt. El video lo demuestra empíricamente. SDAD lo formaliza.

PROMPT RULE — sugerencia estadística CODE GATE — garantía estructural
Falla cuando el modelo no atiende esa parte del contexto Imposible violar — la acción no está disponible

Los dos gaps en detalle

GAP 1 — Minimalismo de herramientas

El video hace del principio "herramientas simples > especializadas" un pilar explícito. SDAD lo practica (skills on-demand) pero no lo articula como principio teórico.

Fix: Agregar en Behavior Rules: "Skills se cargan on-demand precisamente para mantener el prompt base lean — activar solo lo necesario para el increment."

GAP 2 — Umbral de degradación

Video: degradación empieza ~20%, recomienda restart a 40%. SDAD: warning suave a 50%, hard a 65%. Si el video tiene razón, los warnings llegan cuando el daño ya ocurrió.

Fix: Bajar soft warning a 40% o agregar un informational a 30% como "early heads-up". Documentar la decisión de calibración en DECISIONS.md.
