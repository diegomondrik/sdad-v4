Build Discipline

Increment Anatomy

Every increment follows the same contract. An increment that skips any step is not complete — it reopens risk for the next session.

Step 1
Announce
Name the feature, list files to change, tests to run, docs to update, dependencies. Wait for confirmation.
Step 2
Model / Effort
Emit the recommended model + effort. If your session differs, switch before writing any code.
Step 3
Write Code
Write only what the announcement scoped. Expanding scope mid-increment restarts the announcement.
Step 4
Run Tests
Execute the project test command. Report actual results — pass count, failures, errors. Never skip.
Step 5
Run $qa
All 4 layers. Security findings require explicit approval before any fix is applied.
Step 6
Commit
Atomic commit: DECISIONS.md + §13 update + all changed files. One commit per increment.
Compliance

Tier Gates

Tier is confirmed in Phase 1. Once locked, it determines which SPEC.md sections are mandatory and which skills auto-activate.

Tier 1 — Standard
internal · POC
No mandatory gates. Standard Definition of Done. No auto-activated skills.
Tier 2 — Business
SaaS · user data
§9 Security expanded with data classification and retention policy. Compliance Reviewer auto-activates. DoD adds: audit logging, PII documented, auth reviewed, sanitized errors.
Tier 3 — Enterprise
regulated · corporate IT
$build is blocked until SPEC.md §9 is complete and approved. Threat model, data flow diagram, and control matrix are mandatory. Full Compliance Reviewer + regulation-specific skills.
Session Management

Working Across Multiple Sessions

Long projects span many Claude Code sessions. The context budget runs out. SDAD preserves state so the next session picks up exactly where you left off.

At 50% context
Soft Warning
Informational. Finish the current increment, then consider starting fresh after it closes.
At 65% context
Hard Warning
$build is blocked after the current increment. Run $pause compress and start a new session.
$pause compress
Session Snapshot
Generates a compact state block. Paste it at the start of the next session — Claude restores all state without re-explanation.
COMPACT ANCHOR
Lock Survival
PreCompact writes locked decisions to disk. SessionStart re-injects them after compaction. Locked decisions never get lost.
The ccstatusline bar shows context % in real time. Keep it visible — it's the primary context budget indicator for catching the 50% and 65% thresholds before they interrupt work.
Model & Effort Routing

Which Model for Which Phase

SDAD uses tier names — not model names — so the routing table survives model releases. The developer switches manually; Claude announces the recommendation and waits at $build if there is a mismatch.

Tier Examples (2026) Phase Effort Reason
FRONTIER Opus, Fable $spec · $specout high Open decisions, requirements design — needs best judgment
FRONTIER Opus, Fable $qa full · $QA · $docfinal high Whole-codebase judgment — catches what STANDARD misses
FRONTIER Opus, Fable $build — medium/high risk or open decision high Increment has unresolved choice or structural risk
STANDARD Sonnet $build — executing already-specified work low Work is defined; low-risk execution
STANDARD Sonnet $qa incremental medium Bounded review of one increment
ECONOMY Haiku $verify · $doc · $pause · $lesson · $flow low Mechanical, delegable, no judgment required
pinned — $agent review · $agent audit high Pinned in .claude/agents/*.md frontmatter — never overridden
pinned — $agent test medium Pinned in .claude/agents/*.md frontmatter — never overridden
$build es el único que bloquea. Si el modelo activo difiere de la recomendación, Claude lo señala y espera a que cambies antes de escribir código. El resto de las fases notifican una sola vez y continúan.
Document Ingestion

Converting Client Docs Before $spec

The first $spec session on a real project almost always involves client documents — requirements in Word, data dictionaries in Excel, process specs in PDF. SDAD reads Markdown, not binary files. Convert first, then feed to $spec.

Install once
MarkItDown
Microsoft's open-source converter. Preserves headings, tables, and lists — far more token-efficient than raw binary extraction.
Supported formats
PDF · docx · xlsx · pptx · .msg
Also handles images (OCR) and HTML. One command converts any of these to a clean Markdown file.
Output location
.sdad/ingest/
Working copies — not deliverables. Listed in .gitignore. Claude reads from here; the original file stays untouched.
Security rule
Local trusted files only
Never feed untrusted paths or remote URLs — markitdown performs I/O with process privileges. Only convert files the client sent to you directly.
# Install once
pip install 'markitdown[all]'

# Convert each client document
markitdown requirements.docx -o .sdad/ingest/requirements.md
markitdown data_dictionary.xlsx -o .sdad/ingest/data_dictionary.md
markitdown briefing.pdf -o .sdad/ingest/briefing.md

# Then run $spec — Claude reads the converted files automatically
Sub-agent Delegation

What Claude Delegates — and Why

Sub-agents run in isolated context — they don't consume the main session budget and can't see session decisions. Claude delegates automatically when all three conditions are met; no manual instruction needed.

Condition 1
Committed to filesystem
The task operates on files already written to disk. A sub-agent can read them without needing session history.
Condition 2
No session decisions needed
The task doesn't require knowing what was decided during this session — it's self-contained from the code alone.
Condition 3
Expensive in context
The task is large enough to meaningfully consume main-session context: doc generation, architectural review, test suite generation.
Always delegated
  • $doc (all variants)
  • $agent review [module]
  • $agent test [module]
  • $agent audit [path]
Never delegated
  • $qa after $build
  • $spec / $specout
  • $build
Agent files live in .claude/agents/. Output goes to .sdad/agent_output.tmp — read and incorporated silently. If the file is empty or missing, the error surfaces to you; Claude never proceeds on missing agent output.
Decision Log

The HUB BLOCK Format

After every completed increment, Claude generates this block and writes it directly to DECISIONS.md. It is the canonical record of architectural choices — not a narrative, not a summary, but a structured entry that can be queried weeks later.

════════════════════════════════════════════════════════ 📋 HUB BLOCK — DECISIONS_[PROJECT].md ════════════════════════════════════════════════════════ Date: [YYYY-MM-DD] Increment: [N] — [feature name] Model: [exact model string used] Decision: [one-line summary of the main architectural decision] Rationale: [one-line rationale] Alternatives considered: [brief — or "none"] Impact: [files changed, dependencies added, patterns introduced] ════════════════════════════════════════════════════════
Model field
Reproducibility
Records the exact model string (e.g. claude-sonnet-4-6). Lets the team know which capability tier produced which output — critical for audits and regression analysis.
Alternatives considered
Anti-amnesia
Even if "none" — stating it explicitly prevents the team from re-debating a choice that was already made and deliberately rejected.
[LOCK] decisions
Survive context resets
Decisions marked [LOCK] in DECISIONS.md are carried into the COMPACT ANCHOR by $pause compress — they survive context compaction and session restarts.

Build Gates

Three Gates Before $build

Pyplan projects have additional gate sections in SPEC.md. $build is blocked until each applicable gate is approved. The gates exist because building against an unvalidated data architecture is the leading cause of structural delta mid-project.

§A — Data Architecture
always required
Must be complete and approved before $build. Covers: client diagnosis, architecture decision, data contract per source (granularity, refresh cadence, owner, format). §A is filled during $spec before any other section.
§D — MCP Tools Catalog
conditional: @mcp_tool nodes only
Only created when the project declares at least one @mcp_tool node. When present, must be approved before $build. Documents each tool: node identifier, tool name, description, parameter names + types + Annotated descriptions, return type, serialization notes.
§B — Discovery Log
fills during build
Initialized empty at project start. Fills during $build when data deltas are found. Each entry records: assumption from §A, finding, how resolved, and impact on other nodes.

Data Delta Handling

When Reality Differs From §A

Data sources never perfectly match what was assumed at spec time. SDAD classifies every discrepancy as either a small delta (resolve and continue) or a structural delta (pause the build).

Small Delta — continue building

Format errors, nulls, unexpected volume, wrong field name. Resolvable inside the node.

  • Resolve in the node
  • Record in §B: assumption, finding, resolution, impact
  • $build continues without pause
Structural Delta — pause immediately

Data does not exist, wrong granularity, source completely different from §A assumption.

  • Pause $build immediately
  • Generate data gap report: assumption, finding, decision needed
  • Surface to developer — consultant takes report to client
  • $build does not resume until resolution is approved in §B
Never improvise a workaround for a structural delta. A workaround that gets committed becomes technical debt the client discovers during user acceptance testing — at the worst possible moment.
Pyplan MCP

Build-via-AI Guardrails

The Pyplan MCP server allows AI clients to modify application logic and interfaces directly in a running Pyplan instance. SDAD treats each AI-driven modification as an increment — the same rules apply.

Rule 1
Spec gate applies
Spec must be approved before any build/modify action. Pyplan MCP does not bypass the spec gate.
Rule 2
Announce first
Each AI-driven modification is announced as an increment before execution. Wait for developer approval.
Rule 3
QA after every change
Run $qa on the modified increment. No increment is complete without QA — including AI-built interfaces.
Rule 4
Snapshot before commit
After $qa passes, export the model to .sdad/pyplan-snapshots/ before committing. Increment is not complete without it.
Pyplan MCP is a v1 dependency. Document it as an external dependency in §7 and flag its maturity level in $verify. Recommend locking to a specific Pyplan version in §5 if MCP stability is critical.

Build Gate

§E Data Model — Three States

§E is the Board build gate. Unlike Pyplan's binary gate, §E has three states that determine what $build can do.

Empty / Not present
Full block
$build is completely blocked. §E must be at least in Draft state before any code is written.
Draft
Analysis mode
Enables analysis mode for existing projects. Claude can read, audit, and propose changes — but cannot write new Board artefacts. Inferred fields marked [inferred] until developer confirms.
Approved
Full $build
Full $build enabled. All Board artefacts (SQL, CSV, Layout XML, Procedure specs) can be generated and committed.
Project Start

New Project vs. Existing

The first $spec question on Board projects: "New project or existing?" The answer determines the entire onboarding flow.

New project
Guided build
Board questions in order: version, cloud/on-prem, entities, relationships, cubes, capsule structure, procedures, data sources, Board API availability.
Existing project
File ingestion flow
Ingest Layout XML, CFG, CSV, screenshots → auto-populate §E/§F. All inferred fields marked [inferred] — developer confirms each before §E is approved.
Enforced Rules

Entity Creation Order

Board enforces a dependency chain. Creating objects out of order causes silent failures that are hard to diagnose. SDAD blocks out-of-order creation.

1
Entities
Base objects — dimensions and master data
2
Relationships
Links between Entities — requires step 1 complete
3
Cubes
Multidimensional data — references valid Entities only
All Cubes must reference only valid Entities as dimensions. A Cube that references an Entity not yet created, or created in the wrong order, will cause a runtime error that is not always obvious at build time.
Enforced Rules

Algorithm Syntax

Board algorithm formulas have strict syntax rules. SDAD validates every formula before an increment closes.

Block letters
a, b, c… only
Algorithm steps use block letters — never arbitrary variable names. This is a Board platform constraint, not a style preference.
Valid Board functions
dt() rt() gt()
Only Board built-in functions are valid in Algorithm formulas. No Python, no custom functions, no external library calls.
Date references
@DATE @MONTH @YEAR
Use Board's built-in date tokens. No Python datetime imports, no hardcoded date strings in formulas.
Enforced Rules

Procedure Placement

Procedures in Board have a placement rule that determines whether they can be scheduled. Getting this wrong means a procedure that looks correct but can't be automated.

Scheduleable Procedures
must be at Data Model level
Procedures that need to run on a schedule (nightly refresh, month-end close, etc.) must be placed at the Data Model level — never at Capsule level. Only Data Model level procedures can be registered with the Board scheduler.
Client-side Steps
never in server-side Procedures
Navigation steps, selection steps, and UI interaction steps are client-side operations. Placing them inside server-side Procedures causes silent failures — the step runs server-side where there is no UI context to act on.

$audit — Pyplan Audit (v6.0)

Audit Capabilities

Five-dimension judgment of an existing Pyplan model — no SPEC.md required. Evidence-first, not-assessable guarantee, neutrally delivered client report.

When to Use

$audit vs $docfinal

Both commands operate without an approved SPEC.md. The difference is purpose: $docfinal documents what a project does; $audit judges how well it does it and recommends improvements.

Use $audit when
Client-facing judgment needed
A consultant is evaluating an existing Pyplan model before delivery, after an incident, or as part of a due-diligence engagement. The output is a five-dimension report addressed to the client.
Use $docfinal when
Team documentation needed
A project was built without SDAD and now needs a retroactive spec, authorship log, and internal QA report. The output is internal documentation for the development team.
Spec gate: $audit writes its artifacts via the .sdad/AUDIT_ACTIVE sentinel — allowlisted in the spec-gate policy. The sentinel is created on audit start and removed on completion or abort. Security and compliance findings still require explicit developer approval before any fix.
Report Structure

The Five Dimensions

Every $audit report covers these five dimensions in order. Each dimension requires evidence — un-acquirable areas are declared "not assessable," never assumed or fabricated.

1
Development
Code quality, node structure, naming conventions, diagram organization, documentation completeness, formula clarity, error handling, and maintainability.
2
Security P0
OAuth token handling — not logged or exposed in node results. @mcp_tool parameters validated before use — no path to arbitrary code execution. Exposed tools have minimum necessary scope. Data access controls match declared permissions in §A.
3
Usability
Interface completeness, input validation coverage, visualization accuracy, user flow coherence, error messaging quality, and whether the model communicates results clearly to its intended users.
4
Quality
Calculation correctness, index synchronization, circular dependency check, data source validation against declared contracts, test coverage of critical nodes, and consistency of results across scenarios.
5
Business
5a — Alignment: does the model's logic serve its declared business objective? Requires owner elicitation.
5b — Domain Correctness: do calculations follow domain-specific conventions? Requires a matching domain profile (finance, supply-chain).
Evidence First

Evidence Acquisition

Every finding in an audit report must be traceable to acquired evidence. The evidence contract is defined in .sdad/audit/SCHEMA.md — it specifies what can be acquired and how.

Pyplan MCP read
Primary source
Node definitions, result assignments, parameter types, dependency graph, interface structure — acquired via Pyplan MCP when available.
.ppl export
Secondary source
When MCP is not available, a .ppl export analyzed in .sdad/pyplan-snapshots/ serves as the evidence base.
Manual artifacts
Supplementary
Screenshots, business requirement docs, owner interviews, test run outputs — accepted as supplementary evidence for usability and business dimensions.
Not acquirable
"Not assessable"
If evidence for an area cannot be acquired through any method, the area is declared not assessable. This is a finding — it tells the client what could not be verified and why.
The Guarantee

Not Assessable — Never Fabricated

The not-assessable guarantee is a hard rule. When evidence cannot be acquired — or when the required input for a dimension is absent — SDAD declares the area not assessable. It never fills the gap with a plausible-sounding finding.

Business Alignment (5a) — no elicitation input received
"not assessable — no elicitation input"
Business alignment requires owner input: what is the model supposed to achieve, for whom, measured how? Without that input, SDAD cannot assess alignment. A fabricated alignment finding is a liability — it could give false assurance that a misaligned model is strategically sound.
Domain Correctness (5b) — no domain profile for detected domain
"not assessable — no domain profile"
Domain correctness requires a domain profile that encodes field-specific conventions (finance: discount rates, NPV/IRR, FX; supply-chain: lead times, safety stock, demand aggregation). Without a matching profile, SDAD cannot assess whether the model follows the domain's conventions.
Not-assessable is a finding, not a blank. It tells the client: "we tried to assess this, evidence or input was not available, here is what would be needed to complete the assessment." That is more useful — and more honest — than a fabricated result.
Domain Correctness

Domain Profiles

Domain profiles are loaded on-demand when PROJECT_DOMAIN is set. Each profile carries rule-based checks specific to that domain's conventions and common errors.

domain-finance
Financial Models
Checks: discount rate handling, NPV/IRR convention correctness, FX assumptions, period consistency, terminal value treatment, working capital logic.
domain-supply-chain
Supply Chain Models
Checks: lead-time treatment, demand aggregation patterns, safety stock assumptions, MOQ logic, supplier consolidation rules, seasonality handling.
Confidence levels are always stated. An LLM domain profile raises the floor for domain correctness review — it surfaces likely issues a generalist reviewer would miss. It does not replace the client's Subject Matter Expert for high-stakes validation. Every finding carries an explicit confidence level so the client knows what requires SME sign-off.
Report Delivery

Neutrality Requirement

$audit reports are client-facing. They are written for a consultant to hand to a client — often a client whose team built the model being audited. The report communicates intent vs. delivered: here is what the model was likely trying to do, here is what it actually does, here is the gap.

Evidence-based
Only what was found
Every finding cites the specific node, formula, or structure that produced it. No inferences beyond the evidence.
Never accusatory
Intent vs. delivered
"The node calculates X but the declared objective requires Y" — not "the developer made an error." The gap is the finding; the cause is the client's to interpret.
Relationship-aware
Consultant context
The report is a tool for a consultant conversation, not a public indictment. Severity and language calibrate to the professional relationship.
Targeted runs: $audit [dimension] runs a single dimension (e.g. $audit security, $audit business). $audit report generates the final client-deliverable document from completed dimension results.
