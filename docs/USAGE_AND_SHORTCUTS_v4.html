Phase Name What happens Output
Phase 0 Context Analysis Claude reads the repo, detects platform, detects UI presence, recommends compliance tier Tier recommendation, skill suggestions
Phase 1 Requirements Guided discovery — one question at a time with proposed defaults. Covers scope, flows, data model, integrations, security, and compliance tier Requirements captured in session
Phase 2 Spec Document Generates the full 13-section SPEC.md and writes it to the repo root. Requires developer approval before build can proceed SPEC.md written to repo
Phase 3 Build Vertical increments with tests. Each increment announced before coding, tests run after, QA and decisions logged automatically Code, tests, DECISIONS.md entries
Phase 4 QA Layered quality review across security, structure, efficiency, best practices, and platform (Pyplan). Findings numbered H-01, H-02… QA report with findings and fixes

All commands

Core workflow

CommandPhaseDescription
$spec Phase 1 Guided requirements — one question at a time with a proposed default. Reads existing files before asking. Use $spec [section] to jump to a specific section.
$specout Phase 2 Generates the full Spec document from requirements gathered in $spec. Writes SPEC.md to the repo root. Requires approval before $build is allowed.
$build Phase 3 Guided vertical increment. Announces files, tests, docs, and dependencies before writing code. Runs tests after. Use $build [feature] to specify a feature.
$qa Phase 4 Auto QA on the last $build increment. All 4 layers (5 on Pyplan). Findings numbered H-01, H-02…
$qa review Phase 4 Manual QA — presents each finding individually and waits for approval before applying any fix.
$qa full Phase 4 Full project audit across all files, not just the last increment.

Session management

CommandDescription
$pause Show current session state: phase, Spec status, compliance tier, platform, context budget %, decisions log count, open QA findings, active skills, flows, and next recommended step.
$pause compress Generate a compact Session Snapshot. Paste it as the first message in the next session to fully restore state — no need to re-explain anything.
$sdad Show the SDAD v4.3 methodology overview: all phases, descriptions, and command list.

Documentation and verification

CommandDescription
$docfinal Retroactive documentation for projects built without SDAD. Runs 4 steps: retroactive SPEC, AI Authorship Log, standalone QA audit, and lesson candidates. Use $docfinal spec, $docfinal log, $docfinal qa, or $docfinal lessons to run individual steps.
$verify Check dependency documentation currency. Runs automatically when $build introduces a new external dependency. Uses Context 7 MCP when active. On Pyplan projects with §D: always includes the Pyplan MCP server (v1) as an external dependency — flagged in §7.
$doc Generate technical documentation via sub-agent. Variants: $doc readme, $doc api, $doc arch, $doc compliance. All outputs written to /docs.

Session setup

Status bar — once per machine

npx ccstatusline@latest

Interactive TUI: enable the Model, Thinking Effort, Context %, Session Cost and Git Branch widgets and save. This writes statusLine into ~/.claude/settings.json; the bar then renders automatically at the bottom of every Claude Code session.

Every session — Claude Code

claude

Resuming a previous session

If the previous session ended with $pause compress, a Session Snapshot was generated. Paste that block as the first message in the new session — Claude restores all state (phase, Spec status, open findings, decisions, active skills) without any re-explanation.

Context budget

Claude monitors context usage throughout the session and emits warnings at two thresholds.

0% 50% — soft warning 65% — hard warning 100%
ThresholdActionWhat to do
⚠️ 50% Soft warning — session continues normally Consider finishing the current increment and starting a new session. No action required yet.
🔴 65% Hard warning — $build blocked after current increment Finish the current increment completely (including tests and $qa). Then run $pause compress, save the snapshot, and start a new session.
Hard warning never interrupts mid-increment The 65% block activates after the current increment is fully closed, including tests and QA. Commands $pause, $spec, $verify, $lesson, $doc, and $flow remain available even after the hard warning.

Compliance tiers

Tier is detected automatically in Phase 0 based on repo context and confirmed by the developer in Phase 1. Once confirmed, the tier activates additional skills and DoD requirements.

TierForAuto-activatesGates
Tier 1 — Standard Internal tools, POCs, scripts, personal projects Nothing additional None
Tier 2 — Business SaaS, customer-facing products, apps handling user data Compliance Reviewer §9 Security expanded (PII handling, auth review, audit logging)
Tier 3 — Enterprise Regulated environments, cloud to corporate IT, healthcare, finance, government Compliance Reviewer (full) + regulation-specific skill $build blocked until SPEC.md §9 is complete and approved

Sub-agent delegation

Some commands delegate to sub-agents automatically — they run in isolated context and do not consume the main session budget. You see only the final result.

Always delegatedNever delegated
$doc (all variants) · $agent review · $agent test · $agent audit $qa after $build · $spec / $specout · $build
Sub-agent mechanics are silent Agent delegation happens automatically when the conditions are met. You do not need to decide which tasks to delegate — Claude handles it.

HUB BLOCK — decisions log

After every completed $build increment, Claude generates a HUB BLOCK and writes it to DECISIONS.md in the repo root. It captures the key architectural decision made in that increment for future reference.

📋 HUB BLOCK — DECISIONS_[PROJECT].md
════════════════════════════════════════════════════════
Date:                   [YYYY-MM-DD]
Increment:              [N] — [feature name]
Model:                  [model used]
Decision:               [one-line summary]
Rationale:              [one-line rationale]
Alternatives:           [brief — or "none"]
Impact:                 [files changed, dependencies added, patterns introduced]

Lesson Library

After every $qa run, Claude evaluates whether the increment revealed a pattern worth capturing. If it did, one lesson candidate is proposed. You choose to add it, skip it, or edit it. Approved entries are written directly to LESSON_LIBRARY.md.

Lessons are triggered by:

  • A bug or failure pattern likely to recur in other projects
  • An integration quirk not documented in official docs
  • An architectural or prompt pattern that significantly simplified the solution
  • A Pyplan-specific pattern (node design, interface structure, data handling)
Lesson capture is silent when nothing is worth capturing If no lesson-worthy finding is detected after a $qa run, nothing is proposed. The absence of a proposal means the increment was clean — it is not an error.

Flows

Flows are saved sequences of steps specific to this project. Define a flow once, then run it by name in future sessions. Saved to .claude/flows/.

CommandAction

$flow listList all saved flows
$flow [name] runExecute a saved flow
$flow [name] editUpdate an existing flow
