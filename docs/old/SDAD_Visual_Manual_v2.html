01 · The problem

Why SDAD exists

An AI coding agent is, structurally, an eager contractor with no memory between sessions, no inherent notion of project scope, and a strong disposition to be helpful right now — which means it starts coding before anyone has agreed on what "done" means.

The productivity is real: a task that took a day can take an hour, and a non-developer with domain knowledge can direct work they could not previously execute. But the failure modes are real too, and they are different from human-only development. Five of them recur across every team that adopts AI coding tools without a methodology:

Failure modeWhat it looks like
Premature implementationThe agent codes immediately. By the time the scope error is noticed, there are hundreds of lines of plausible code solving the wrong problem.
Silent assumptionsReasonable-sounding guesses about data shapes and edge cases get baked into the build, with no record of what was assumed or why.
Session amnesiaWhen the context window expires, prior decisions are gone. The developer re-explains everything, or the agent drifts back to defaults.
Governance driftPrompt rules ("always run tests", "never touch production") degrade over long sessions. A rule obeyed on turn 10 is quietly forgotten on turn 40.
No auditabilityNo record of what was built, why, what was rejected, or which AI model made which decision.
In one breath AI writes code fast, but left to itself it builds before it understands, forgets what was decided, and treats your rules as polite requests. SDAD is the discipline that fixes this — so AI-assisted work becomes predictable, auditable, and safe for non-developers to direct.
Why a methodology, not just better prompts The instinctive fix is to write better prompts. It helps, but it has a ceiling: a prompt is a natural-language suggestion to a probabilistic system, not a constraint. SDAD's answer is structure — enforced phases, persistent state, and hard gates for the rules that must never be violated.
The engineering claim Governance-by-prompt provides statistical compliance; governance-by-code provides guaranteed compliance. SDAD's thesis is to use both layers in their correct domain — code for rules with serious consequences and a mechanically detectable trigger; prompt for guidance and judgment calls. Section 4 makes this concrete.

One-sentence definition SDAD is a methodology that imposes process discipline on Claude Code through a structured phase model, a command surface, automated session lifecycle management, a tiered compliance system, a cross-project Lesson Library, and — in v5 — hard enforcement of critical rules in executable code rather than natural-language instruction.

The gates are the point. Phase 3 (Build) cannot begin until Phase 2 produces an approved specification. Phase 4 (QA) runs automatically after every Phase 3 increment. The sequence is not self-reported — it is enforced.

Read the picture left to right First SDAD reads your project. Then it asks what you want (one question at a time). Then it writes that down as a contract you approve. Only then does it build — in small, tested slices — and it checks quality after every slice.
▸ How it works — what each phase does technical layer
  • Phase 0 — Context Analysis. Before asking anything, SDAD reads files, git history, manifests, and any prior SPEC.md/DECISIONS.md. It determines new-vs-resume, the stack, whether a UI exists, the likely compliance tier, and whether a $verify audit is due (>30 days since last build). It also surfaces 2–3 relevant lessons from the Lesson Library.
  • Phase 1 — Requirements ($spec). A Socratic dialog: one question at a time, each with a proposed default. First question is always project language. The compliance-tier question is mandatory and cannot be skipped.
  • Phase 2 — Specification ($specout). Generates the full 13-section SPEC.md and writes it to the repo root. This is the contract; $build stays locked until it is approved.
  • Phase 3 — Build ($build). Implements one vertical increment per call — a complete, tested slice from user interaction down to data, not a horizontal layer.
  • Phase 4 — QA ($qa). Runs automatically after every increment, in four priority-ordered layers (Security → Structure → Efficiency → Best Practices).

03 · The walkthrough

A project, start to finish

The same five phases, but now with the real artifacts SDAD produces at each step — the spec contract, the increment announcement, the decision log, the QA pass.

Phase 1 — the Socratic dialog

The first question is always language: should interaction and documents be in English or Spanish? The answer becomes PROJECT_LANGUAGE and governs everything after (code identifiers and comments always stay in English). The standard sequence then covers scope, user roles, data model, integrations, business rules, performance, security & compliance (with mandatory tier confirmation), and testing strategy.

Phase 2 — the spec is a contract

$specout writes a complete 13-section SPEC.md. It is not documentation of what was built — it is a commitment to what will be built, agreed in advance by both the developer and the AI. Section §13, the AI Authorship Log, is a typed 8-column table (Increment, Feature, Model, Effort, Files, Tests, QA Findings, Date) that makes every increment auditable.

Phase 3 — every increment announces itself

Before writing a single line, $build announces the increment. If the active model does not match the recommendation, it flags the mismatch and waits:

🔨 INCREMENT 3: customer checkout flow 🧠 MODEL: STANDARD · effort low — mechanical implementation (recommended: use /model and /effort if the active session differs) Files: src/checkout.py, tests/test_checkout.py Tests: unit + integration — will be executed after writing Docs: README update + inline comments Dependencies: cart module (increment 2) ────────────────────────────

After writing code, $build runs tests (reporting real pass/fail counts, never simulated), runs $qa, writes the DECISIONS.md HUB BLOCK, and updates §13 — the tests, decision, and log committed atomically as one unit.

The decision log — a real HUB BLOCK

════════════════════════════ 📋 HUB BLOCK — DECISIONS.md ════════════════════════════ Date: 2026-06-15 Increment: 3 — customer checkout flow Model: STANDARD Decision: use idempotency keys on the charge endpoint Rationale: retries must not double-charge the customer Alternatives considered: dedup table (heavier, rejected) Impact: src/checkout.py; adds stripe idempotency dependency

Every architectural decision becomes a searchable record, attributable to a model, date, and increment. This is the audit trail that makes AI-assisted development accountable.

What you'll actually see You answer a short interview, you read and approve a spec, and then the AI builds in small pieces — announcing each one before it starts, testing it, and writing down every decision. Nothing important happens silently.
Where you stay in control Three checkpoints are yours: you approve the spec before any build, you confirm the model tier when an increment announces a mismatch, and you approve any security or compliance fix. The vertical-increment rhythm means you review small, complete slices — never a 2,000-line drop.
▸ How it works — the post-write sequence & state files technical layer

After writing code, $build runs, in order: (1) the project's test command, reporting real results; (2) $qa on the increment; (3) a DECISIONS.md HUB BLOCK; (4) the §13 update; (5) on Pyplan, the platform checklist; (6) after a structural increment, a proposed CLAUDE.md update. Steps 3–4 plus the tests commit atomically — a v5 discipline so a mid-write crash never leaves inconsistent state.

SDAD's permanent project memory lives in three git-backed files: SPEC.md (the contract), DECISIONS.md (the decision log; [LOCK] entries are non-reopenable), and LESSON_LIBRARY.md (cross-project knowledge). Sub-agents handle expensive, committed-file work ($doc, $agent review/test/audit) in isolated context that doesn't consume the main session budget.

Context budget is watched at two thresholds: 50% soft warning (continue), 65% hard warning (finish the current increment cleanly, then block new builds until $pause compress snapshots state for a fresh session).

03b · Memory

Memory & persistence between sessions

An AI agent has no memory of its own — the context window empties when the session ends, and compaction can drop detail mid-session. SDAD treats memory as an engineered layer, not a hope. It persists what matters in four distinct tiers, each with a different lifetime and a different home.

Lifetime: this session
Layer 1 · Within-session context
The working context window

Everything the agent currently "sees" — the conversation, files it has read, decisions made this turn. Fast but volatile: it degrades as it fills and is lost when the session ends. SDAD does not trust it as memory; it watches it with the context budget (50% soft / 65% hard) and parks work cleanly before it runs out. Sub-agents run in isolated context so heavy work never crowds this window.

Lifetime: across sessions, this project
Layer 2 · Cross-session project files
SPEC.md + DECISIONS.md + the anchor

The durable project memory, all git-backed in the repo. SPEC.md holds the contract; DECISIONS.md holds the decision log (with [LOCK] entries that are non-reopenable). The COMPACT ANCHOR — the set of [LOCK] decisions and live constraints — is written to disk by the PreCompact hook and re-injected by SessionStart, which is what makes it survive both compaction and a full restart. $pause compress produces a Session Snapshot: a compact state block you paste into the next session so SDAD restores phase, tier, approved sections, open QA findings, and the exact next step — with no re-explanation.

Lifetime: forever, every project
Layer 3 · Cross-project knowledge
The Lesson Library

LESSON_LIBRARY.md is the only memory that crosses project boundaries. A lesson learned on one project is surfaced automatically in Phase 0 of every future project, matched by #stack and #phase tags (embeddings only past ~50 entries). This is how the methodology compounds: a mistake made once becomes a guardrail everywhere — and, when mechanically checkable, ratchets into code (Section 8).

Deliberately not persisted
Layer 4 · What is NOT persisted
The forgetting is also a design choice

Conversational chatter, intermediate reasoning, rejected drafts, and a sub-agent's full working context are intentionally not saved — only its HANDOFF result returns to the main session. Nothing already recorded by the code itself, git history, or SPEC.md is duplicated into memory. The rule is: persist decisions and contracts, discard the deliberation that produced them. Keeping everything would poison the context window it is meant to protect.

Why you never re-explain The AI forgets the conversation when a session ends — but SDAD doesn't. The plan, the decisions, and the lessons are written to files in your project, so the next session picks up exactly where you left off. And lessons from other projects show up automatically when they're relevant.
The four tiers in one line each Session context (volatile, watched by the budget) → project files (SPEC + DECISIONS + anchor, git-backed) → the Lesson Library (cross-project, surfaced in Phase 0) → and a deliberate discard layer that keeps the working window clean.
▸ How it works — the survival path & the snapshot technical layer

Anchor survival is a two-step mechanism, by design. A [LOCK] decision: PreCompact's own context injection does not survive compaction — so survival = PreCompact writes the anchor to disk + SessionStart re-injects it after compaction or restart. Either half alone fails.

The Session Snapshot ($pause compress) carries: active phase / tier / platform, approved spec sections, the active increment, the [LOCK] decisions only (not the full decision log), open QA findings (H-XX), the Lesson Library delta, context budget %, and the exact next step. When SDAD detects a snapshot at conversation start, it restores all state without asking the developer to re-explain.

  • Volatile — the context window. Protected, never trusted.
  • Durable, per-project — SPEC.md, DECISIONS.md, COMPACT ANCHOR, Session Snapshot.
  • Durable, cross-project — LESSON_LIBRARY.md, surfaced in Phase 0.
  • Discarded — deliberation, sub-agent working context, anything already in code or git.

Source — thesis §4.2 (state model), §4.5 (context budget), §6.4 (PreCompact / SessionStart anchor), §8 (Lesson Library).

04 · The big idea

Governance by code, not by prompt

SDAD's whole reason for existing fits in one sentence: a rule written in a prompt is a suggestion; a rule written in code is a guarantee. Everything else in this manual is machinery built to honour that distinction.

A rule in a prompt is a suggestion.
A rule in code is a guarantee.
PROMPT RULE — statistical suggestion CODE GATE — structural guarantee
Fails when the model stops attending to that part of context Impossible to violate — the action is not available

SDAD places each rule on this spectrum deliberately. "Ask one question at a time" and "always propose a default" are judgment calls with no mechanical trigger — they stay on the left, as prompt guidance. The spec gate, the ASCII-PS1 check, and atomic state commits have a binary, detectable trigger — they ratchet to the right, into code. The governance axiom is choosing the correct end of the bar for each rule.

In one breath Think of an eager new contractor who is brilliant but forgetful. You can ask them to never touch production without sign-off — and they will, until the day they are busy and forget. Or you can take away the key. SDAD takes away the key: the most important rules are enforced by the tooling itself, so "we forgot" stops being a possible outcome.
How it flows in a project On your projects this means the AI literally cannot start building before there is an approved spec. Not "is told not to" — cannot. The spec you approve becomes a contract the tooling enforces, so scope is locked before a line of code exists, and every later decision is traceable back to it.
The mechanics The critical gates live in a PreToolUse hook and in checks/ + a git pre-commit hook — not in CLAUDE.md prose. A Write/Edit call with no approved SPEC.md exits 2 before the file is ever touched. Expand the technical layer below for the exact gate logic.

What happens under stress

Turn 60 of a long session. The model is distracted, the context is full. Now what?

Rule lives in the prompt

"Never write code before an approved spec."

  • Obeyed only while the model is attending to that instruction
  • Degrades over long sessions and after compaction
  • Compliance is statistical, not guaranteed
  • When it fails, it fails silently — code already written

Result: usually fine. Occasionally, 300 lines solving the wrong problem.

Rule lives in code

A PreToolUse hook checks for an approved SPEC.md.

  • The forbidden action is simply not available
  • Independent of the model's state or turn count
  • Compliance is structural — true by construction
  • When blocked, it blocks loudly, with a clear next step

Result: "code before approved spec" is impossible, not unlikely.

See it happen — a real spec-gate denial

# No approved SPEC.md exists yet. The agent tries to write code: > Write src/payments.py ✗ BLOCKED by spec gate (PreToolUse hook, exit 2) SPEC.md is absent or not marked "SPEC STATUS: APPROVED". No production code may be written before an approved spec. → Run $spec to define requirements, or $docfinal for existing code. # After $spec / $specout and developer approval: > Write src/payments.py ✓ allowed — SPEC STATUS: APPROVED found

The gate reads the file system, not the model's intentions. Markdown, docs, and .sdad/ paths are always allowed — only production code is gated.

▸ How it works — the gate logic technical layer

The spec gate is a PreToolUse hook registered in .claude/settings.json. It intercepts every Write and Edit call before execution and runs four steps:

  1. Extract the target file path from the tool call.
  2. If the path is on the allowlist (.md files, docs/, .sdad/, .claude/, hub/, or any path when the .sdad/DOCFINAL_ACTIVE sentinel exists) — allow unconditionally.
  3. If SPEC.md is absent or lacks the exact string SPEC STATUS: APPROVED — deny with exit 2 and a message pointing to $spec or $docfinal.
  4. Otherwise — allow.
Fail-open by design. If the gate script itself errors, it allows the action and logs a warning to .sdad/gate.log. A broken guard must never freeze the developer — its job is to block unauthorized writes, not to gate-keep all writes.

The hook ships as both .ps1 (Windows/PowerShell, tested) and .sh (bash/macOS, pending the macOS test gate), dispatched by a cross-platform run-hook.sh.

05 · The theory

The harness model — H = (E, T, C, S, L, V)

A "harness" is the software layer that governs an AI agent's execution — the term is borrowed from a wiring harness that constrains and channels electrical connections. A well-designed harness decomposes into six governance functions. SDAD uses this map to decide which rules belong in code and which can stay as guidance.

The popular mental model puts the AI model in the centre with four things around it — Context, Tools, Memory, and Validation. That picture is correct but incomplete: SDAD's six-function decomposition (below the diagram) adds the two the four-part picture leaves implicit — the Execution loop (E) and the Lifecycle hooks (L). The diagram maps the familiar four onto SDAD's letters and shows where the extra two live.

E

The observe-think-act cycle; error recovery and loop termination.
SDAD: E-termination protocol
T

Typed, validated boundary between model and environment.
SDAD: PreToolUse spec gate
C

Compaction, retrieval, prioritization within the token budget.
SDAD: 50%/65% budget, $pause
S
State Store
Memory that survives turns and sessions, committed safely.
SDAD: SPEC + DECISIONS + LESSON
L
Lifecycle Hooks
Interception, approvals, audit trails, heartbeat detection.
SDAD: four session hooks
V
Evaluation Interface
Instrumentation that captures what the agent did and whether it worked.
SDAD: $eval golden dataset
The governance axiom Structural constraints are non-negotiable; natural-language pleading is merely a suggestion. A rule in a prompt is obeyed only while the model is compliant, attending, on that turn. A rule in code is obeyed by construction — because the action it forbids is not available.
Why six letters matter to you You don't need to memorize them. The point is that SDAD didn't invent rules ad hoc — it used an engineering framework to decide, deliberately, which safeguards are strong enough to be guarantees and which remain good advice.

07 · Safety

Quality, compliance & safety

QA in SDAD is not a post-hoc check — it is Phase 4, an integral step of every increment, run in four priority-ordered layers. Security goes first, because a critical finding there can make the other layers irrelevant.

#LayerFocusExample findings
1SecurityAPI-key exposure, injection, PII, authHardcoded keys, credentials in logs, unprotected endpoints
2StructureArchitecture consistency, coupling, error handlingMissing error recovery, tight coupling
3EfficiencyToken usage, redundant calls, latencyUnbounded loops, repeated LLM calls
4Best PracticesReadability, naming, documentationUndocumented interfaces, duplication

Findings are classified Must fix (blocks the increment), Should improve (your decision), or Style suggestion (noted). A fifth Platform layer runs on Pyplan projects.

Security never auto-changes Layer 1 (Security) and compliance findings always require explicit developer approval before SDAD applies any fix. $qa auto mode never silently edits security or compliance code. Critical issues are marked 🚨 and compliance violations 🔒 — regardless of the current phase, and never suppressed.

The three compliance tiers

SDAD recommends a tier in Phase 0 from repo signals, you confirm it in Phase 1, and it then applies tier-appropriate rigor throughout. Each step up adds rigor:

Tier 1 — Standard
Internal tools, proofs of concept, scripts. No extra Definition-of-Done requirements beyond the standard QA layers.
Tier 2 — Business
Customer-facing products, SaaS, any project handling user data. Auto-activates the Compliance Reviewer. Adds audit logging, PII handling, auth review, sanitized errors.
Tier 3 — Enterprise / Regulated
Cloud, healthcare, financial, government (ISO 27001, SOC 2, GDPR, HIPAA). Adds threat model, data-flow diagram, control matrix. $build is blocked until §9 is complete and approved.
The risk story Quality is checked after every slice, not at the end. Higher-stakes projects (customer data, regulated industries) automatically get more rigor — and for the strictest tier, the AI cannot build at all until the security section is signed off.
The MCP-vs-CLI lock Layer 1 also enforces a [LOCK] rule: if a CLI wrapper was chosen over a vetted MCP (§7), the wrapper must not introduce shell injection, credentials-in-argv, or fragile parsing risk — and this check cannot be waived on token-cost grounds. Tier-3's §9 gate is structural, like the spec gate: no code until the compliance section is approved, regardless of what the model is otherwise willing to do.

07b · Theory meets practice

Three pillars — harness theory meets SDAD practice

The harness idea is often summarized as three operational pillars: the repo is the system, work is orchestrated across agents, and results must be verified rather than declared. SDAD implements all three — with more formality at each. Here is the theory beside the practice.

📁
1 · The repo is the system
The theory: the whole harness lives in repo files — an agents file, a feature list, a progress folder. The model reads files to know what to do; its context is built from the repo, not from chat.
SDAD in practice
SPEC.md — a 13-section contract, not just a task list.
DECISIONS.md — a HUB BLOCK per increment, with [LOCK].
LESSON_LIBRARY.md — cross-project knowledge.
§13 Authorship Log — attributable by model and date.

More structured and more auditable than a feature list. The spec is a contract, not a to-do.
🤖
2 · Multi-agent orchestration
The theory: a lead agent dispatches specialist sub-agents in isolated context, which write results to a shared folder — avoiding the "telephone game" of passing one bloated context around.
SDAD in practice
Isolated sub-agents for $doc, $agent review/test/audit.
.sdad/lib/agent-run — 600s timeout + empty-output guard.
HANDOFF BLOCK — a structured return per sub-agent.
Model/effort routing — FRONTIER / STANDARD / ECONOMY per phase.

SDAD adds zombie-agent detection and explicit model routing — the theory names neither.
✅
3 · Verification
The theory: the agent must demonstrate that something works (tests, browser checks), never merely claim it. The reviewer may even rewrite its own instructions to self-improve.
SDAD in practice
$qa runs automatically after every $build — 4 layers.
Real tests — never simulated; real pass/fail counts reported.
$eval golden dataset — validates the methodology itself.
Governed self-improvement — the agent proposes a CLAUDE.md update (step 5.5); the developer approves.

A philosophy difference: the theory allows autonomous self-modification; SDAD keeps the developer in the loop.
The honest comparison The popular three-pillar picture is a good map — and SDAD follows it. The difference is rigor: a contract instead of a task list, guarded sub-agents instead of loose ones, and a methodology that tests itself. The one place SDAD deliberately diverges is letting the AI rewrite its own rules: SDAD makes it propose, and a human approves.
Where SDAD goes beyond all three pillars None of the three pillars covers compliance tiers, the §13 audit trail, cross-project lesson capture, model/effort routing, or methodology self-evaluation ($eval). These are SDAD-specific additions, not implementations of the harness theory — they make AI-assisted work accountable, not just controllable.

"SDAD imposes a contract before code begins, records every decision while it is built, and verifies every result when it ends — all inside the repository, all attributable to a model and a moment."

08 · Memory across projects

The learning system & the ratchet

Most development knowledge is siloed in individuals: one developer learns an expensive lesson, then the next person learns it the same expensive way. SDAD's Lesson Library breaks the silo — a lesson captured on any project becomes available on every future project, surfaced automatically in Phase 0 by stack and phase.

The two-layer ratchet

A lesson starts as prose. When its pattern becomes mechanically checkable, it migrates into code — and once in code, it cannot be forgotten, misremembered, or skipped on a particular turn.

QA finds a patternlikely to recur
→
Lesson L-XXwritten to the library (prose)
→
Checkable?binary, mechanical test
→
Check in checks/wired to session-end + git pre-commit
→
Cannot recurblocked by construction
The story that proves it — L-01 "PowerShell hooks must be pure-ASCII." This lesson was captured, written to the library — and then recurred anyway. That recurrence is the proof prose ratchets fail. It triggered the migration: L-01 is now enforced by checks/ascii-ps1 and a git pre-commit hook, not only a prose rule. The pattern that produced this mechanism is the same pattern the mechanism now prevents.
Why this compounds The methodology gets safer every time it is used. A mistake made once on one project becomes a guardrail on all projects — and the most important guardrails get turned into code that simply can't be bypassed.
▸ How it works — lesson structure & the v5 entries technical layer

Each L-XX entry carries: Category (LLM Design, Architecture, Data & Debugging, Environment, Workflow, Pyplan), Tags (#stack:<tech> + #phase:<phase> for filtered retrieval), a Signal (how another developer recognizes it applies), a Principle (one transferable sentence), and an Origin. Capture is silent when nothing is worth capturing — SDAD never forces trivial entries.

  • L-01 — PowerShell hooks must be pure-ASCII and handle UTF-8 explicitly (ratcheted into code).
  • L-02 — Validate single-source rules against real workflows: the premise can be wrong even when the rule is right.
  • L-03 — PS 5.1: native-command stderr under ErrorActionPreference Stop becomes a terminating error.
  • L-04 — Reference CLAUDE.md by its exact case on case-sensitive surfaces (ratcheted into checks/claude-md-case).

Migration from Layer 1 (prompt) to Layer 2 (code) is triggered by recurrence: if a lesson is violated despite the prose rule, that is evidence the prose ratchet failed and the pattern needs code enforcement.

09 · Platform track

The Pyplan track (relevant only to some readers)

▸ Expand the Pyplan extensions platform-specific

Pyplan is a Python BI and financial-modeling platform that organizes computation as an influence diagram (a DAG of node functions). Its projects have unique characteristics — data from client sources the team doesn't control, computation in a visual node graph, and platform-specific interface constraints — so SDAD adds extensions that activate automatically when PROJECT_PLATFORM: pyplan is declared in §0.

§A — the Data Architecture gate

In conventional projects the team defines the schema. In Pyplan, data comes from the client (spreadsheets, ERP, accounting files) and the architecture must fit what actually exists. §A requires a client diagnosis, an architecture decision, and a data contract per source. §A is a gate: $build is blocked until it is approved — because discovering a data mismatch after building model nodes is expensive, while discovering it first is nearly free.

Small delta vs. structural delta A small delta (wrong field name, unexpected nulls) is resolved in the node and logged in §B; the build continues. A structural delta (data doesn't exist, wrong granularity) pauses $build immediately and produces a data-gap report for the client — because an improvised structural fix produces a model that looks like it works but is built on a wrong foundation.

§D — the MCP Tools Catalog gate

When a project exposes model nodes as MCP tools (the @mcp_tool decorator), §D documents each tool (identifier, description, Annotated parameter schema, return serialization). §D is a gate too, and the increment checklist adds six MCP-specific items (docstrings, typed params, serializable returns, result = _fn not _fn(), no session-state dependence, §D updated).

HTML interfaces & Build-via-AI

Each HTML interface is its own increment closed with a dedicated checklist (all traffic through window.pyplan.callback, no cookies/localStorage, model writes via set_input/set_form_values). Build-via-AI — Pyplan's natural-language edit capability — is permitted with full SDAD discipline (spec gate + increment + approval + QA) rather than blocked, so it stays auditable instead of pushing developers outside oversight.

Source — thesis §9.1–9.7 (Pyplan extensions), Appendix B (§ structure).

10 · What makes v5 different

The v5 enforcement layer

From v4.0 to v4.3, SDAD governed almost everything through CLAUDE.md instruction. v5 is a major version because it changes SDAD's identity: from a methodology governed by instruction to one governed by instruction plus enforced code. Four hooks, a self-evaluator, and a defined error state do the work.

Four session hooks

HookWhat it does
SessionStartRe-injects the COMPACT ANCHOR ([LOCK] decisions); runs git pull --ff-only if the tree is clean; reminds you if CLAUDE.md changed since the last $eval pass. Never blocks startup.
PreCompactWrites the COMPACT ANCHOR to disk before compaction — because PreCompact's own context injection does not survive compaction. Survival = write-to-disk + re-inject at SessionStart.
SessionEndAutocommits a whitelist (DECISIONS.md, LESSON_LIBRARY.md — never code). Respects the HOLD_AUTOCOMMIT sentinel; runs the ascii ratchet check; never commits to main directly or commits empty.
PreToolUseThe spec gate (Section 4): blocks Write/Edit without an approved SPEC.md.

$eval — the methodology evaluates itself

Before v5, SDAD edited itself every release but had no way to detect whether an edit introduced a regression — regressions surfaced only when a real project later failed. $eval is the V component: a deterministic golden dataset of 13 scenarios (gate denies/allows, fail-open, ascii check, pre-commit block, regression detection, timeout, typed §13, and more) plus a release-only LLM smoke test.

The plain version v5 is the version where the most important rules stopped being polite requests and became code that can't be ignored. SDAD also gained the ability to test itself before any change reaches a real project — the kind of check most teams never apply even to their production software.
What this buys the team Decisions survive session restarts and context compaction (COMPACT ANCHOR + [LOCK]). Errors stop cleanly instead of looping (E-termination). And a change to the methodology itself is caught by $eval before it can break a client project.
▸ How it works — E-termination, atomic commits & the [LOCK] registry technical layer

E-termination protocol. On a tool/test error mid-increment, SDAD reports clearly, stops cleanly, creates .sdad/HOLD_AUTOCOMMIT (suppressing SessionEnd autocommit), and does not retry or loop. This converts an open-ended ReAct-style runaway into a clean, visible stop. Recovery clears the sentinel and resumes.

Atomic state commits. Tests, the DECISIONS.md entry, and the §13 update commit as one unit, so a mid-write crash never leaves inconsistent state.

The five [LOCK] decisions (non-reopenable, carried in every anchor): (1) MCP-vs-CLI security is a hard gate, never a cost choice; (2) CLAUDE.md stays lean; (3) net line budget ≤ +60 per release; (4) anchor survival = PreCompact-writes-to-disk + SessionStart-re-injects; (5) no API dependency in the methodology.

Source — thesis §6.1–6.7 (the harness layer), Appendix E (the [LOCK] registry).

11 · Getting started

Setup & daily use

SDAD installs once into your Claude Code configuration, then each project repo is initialized separately. Day to day, the rhythm is: open Claude Code, $spec, approve, $build, repeat.

The daily loop

claude # open Claude Code in the project directory $spec # guided requirements (Phase 1) [answer questions] $specout # generate SPEC.md (Phase 2) [review and approve SPEC.md] $build # first increment (Phase 3) -> tests -> $qa [review HUB BLOCK, confirm any lesson capture] $build # next increment...
The short version Install once, then for each project you run a short interview, approve a plan, and let the AI build in checked steps. A status bar shows how much "memory" the session has left so work is parked cleanly before it runs out.
▸ How it works — install, init, resume, verify technical layer

Prerequisites: Claude Code, Node.js 18+, Git. Windows: PowerShell 5.1+ and Git for Windows. macOS/Linux: bash. Strongly recommended: ccstatusline (npx ccstatusline@latest) — the real-time status bar showing model, effort, context %, cost, and branch. It is the primary context-budget indicator, showing the 50%/65% thresholds live.

Install runs a one-line installer (PowerShell on Windows, curl | bash on macOS/Linux) that checks/installs Node and Claude Code, copies CLAUDE.md, installs skills/agents, registers all four hooks, scaffolds checks/ and .sdad/eval/, and installs the git pre-commit hook. project-init then creates SPEC.md, LESSON_LIBRARY.md, DECISIONS.md, and .sdad/ in the repo.

Resuming a session: SessionStart re-injects the anchor and runs a clean fast-forward pull; if a $pause compress snapshot exists, paste it and SDAD restores state with no re-explanation. Verify with $sdad (version 5.0), $skills, $spec (first question = language), $pause (context %), and $eval (13 scenarios all green).

13 · The road ahead

Where it's going — v6 "Team & Portfolio Edition"

SDAD v5 solved the single-developer governance problem with unusual rigor. The next frontier is the deployment G7 is actually heading into: many clients, many projects, many developers on mixed operating systems. The good news is that the mechanisms v5 invented are exactly the right ones — the work ahead is relocation, not reinvention.

The one-line thesis for v6 A gate that lives on the developer's machine is a suggestion; a gate that lives in the shared pipeline is a guarantee — and a methodology used across many clients must be governed from one place, or it is not governed at all.

v5's gates live in local hooks, which means the headline guarantee holds on one correctly-configured machine and weakens once a second developer, a second OS, or a CI server enters the picture. v6's job is to move the same mechanisms into the shared pipeline and a central, multi-project home.

Two honest gaps vs the harness theory

A comparison against the harness-engineering literature surfaced two places where the theory points at something SDAD handles less crisply than it could. Neither is an architecture flaw — both are documentation or calibration opportunities, and both are tracked openly here.

GapWhat the theory arguesWhere SDAD standsStatus
Tool Minimalism Stripping specialist tools from an agent measurably improves it — one reported case removed ~80% of bespoke tools and got faster, cheaper runs. The principle: a more complex harness performs worse. SDAD already practices this — skills load on-demand so the base prompt stays lean — but never articulates minimalism as a stated performance principle. The 10+ specialist skills can read as the opposite intent. Gap (minor) — articulate the principle: "activate only the skills the increment needs."
Context Threshold Context quality is reported to degrade early — noticeable around 20%, with cleanup advised by ~40% of the window. SDAD's soft warning fires at 50%, hard at 65%. If early degradation is real, the first warning arrives after some quality has already been lost — most visible in long $build sessions. Gap (calibration) — consider a 30–40% early heads-up, or document the empirical basis for 50%.
For balance — where SDAD already exceeds the theory The same comparison found SDAD goes beyond the harness theory on the spec gate (no code without an approved contract), compliance tiers, the §13 audit trail, cross-project lesson capture, model/effort routing, and methodology self-evaluation ($eval). The two gaps above are the honest exceptions, not the rule.

The prioritized roadmap

PriorityMove

P0Cross-platform parity — finish and test the .sh gate on macOS/Linux; run $eval on a CI matrix across all three OSes.

What this means for the team The strong guarantees you get on one machine today will become strong guarantees across every client and every developer. v6 isn't a rewrite — it takes the safeguards that already work and installs them in the shared pipeline and a central home.

14 · Reference

Glossary

The terms that most often block non-technical readers — defined plainly, in the order you meet them.

Spec (SPEC.md)
The specification document SDAD writes and you approve. A contract for what will be built, agreed before any code exists.
Increment
One small, complete, tested feature — a vertical slice from user interaction down to data, not a horizontal layer.
Hook
A small script the tooling runs automatically at a key moment (session start/end, before a file write, before compaction).
Gate
A hard checkpoint that blocks progress until a condition is met — e.g. no build before an approved spec.
Harness
The software layer that governs an AI agent's execution. SDAD models it as H = (E, T, C, S, L, V).
Ratchet
The one-way migration of a lesson from prose into enforced code, so a fixed failure mode cannot recur.
CI
Continuous Integration — an automated server that runs checks on every code change. Where v6 will move the gates.
MCP
Model Context Protocol — a typed, validated way to expose tools (e.g. Pyplan model nodes) to an AI client.
Compliance tier
The rigor level of a project: Tier 1 (standard), Tier 2 (business / user data), Tier 3 (regulated).
Context budget
How much of the session's working memory is used. SDAD warns at 50% and blocks new builds at 65%.
HUB BLOCK
The structured decision record written to DECISIONS.md after every increment — date, model, rationale, alternatives, impact.
COMPACT ANCHOR / [LOCK]
The set of non-reopenable architectural decisions that survive session restarts and context compaction — written to disk by PreCompact and re-injected by SessionStart.
Session Snapshot
The compact state block produced by $pause compress — phase, tier, approved spec sections, open QA findings, lessons, and the exact next step — pasted into the next session so SDAD restores state with no re-explanation.
Tool Minimalism
The principle that fewer, simpler tools make an agent faster and more reliable than many specialist ones. SDAD practices it by loading skills on-demand to keep the base prompt lean.
§13 / AI Authorship Log
A typed 8-column table in the spec recording which model made which decision, at what effort, with what QA result.
Fail-open
A safety choice: if a guard script itself breaks, it allows the action (and logs a warning) rather than freezing the developer.
Sub-agent
A delegated AI run in isolated context for expensive work (docs, review, audit) that doesn't consume the main session.
Pyplan
A Python BI / financial-modeling platform; SDAD has a dedicated extension track for it.

Source — terms drawn from thesis §3, §4, §6, §8, §9, and the appendices.
